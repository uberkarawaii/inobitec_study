## Windows, Linux, Android и др
Таненбаум цитата: "...Linux, IOS, Andriod - производные от UNIX" 
как родился *UNIX*:
1ая ОС с разделением времени (CCTS) -> MULTICS -> человек из MULTICS нашёл PDP7 и написал под него 
однопользовательскую версию MULTICS -> UNIX
от UNIX пошло много версий ОС. чтобы была совместимость - POSIX стандарт библиотченых функций для syscall-ов. 
сами syscall-ы могут меняться, а ввиду POSIX будет единообразие обращений к ним

UNIX и Windows имеют разные стандарты системных вызовов

от UNIX -> MINIX -> *Linux*
*Linux*
- имеет трёхслойную структуру над аппаратным обеспечением
  упрощённая схема: *cli / giu -> библиотека syscall-ов (POSIX) -> ОС (настоящие syscall)* -> АО
- графический интерфейс наложен поверх этих слоев. то есть пользователь
  может общаться с библиотекой сист. вызовов либо через терминал, либо через GUI
  популярная GUI: GNOME; популярная оболочка (то, что релизует cli): bash 

*Android*
упрощённая схема: приложения -> API android -> системные службы -> android runtime -> Linux ОС
- ОС не с нуля, а основанная на Linux. бОльшая часть Android написана на Java потому ООПшная
- в Android есть понятие системных служб. это фоновые процессы, которые взаимодействуют с системой Android
  в то время как приложения общаются именно с этими службами через API android. никакое приложение не общается с Linux
- каждое приложение - это процесс, который живёт в своей среде выполнения (android runtime)
  так память разделяется, проблемы одного процесса не трогают другой процесс
широко распространено на мобильных устройствах

*Windows desktop*
упрощённая схема: приложения -> windows API -> системные службы и сервисы -> ядро виндовс (NTOSKRNL) -> hardware abstraction layer (HAL) -> АО
- в ядро встроен GUI (win32k.sys) то есть GUI будет загружаться всегда 
- т.к. win32k.sys в ядре, то оно весит больше чем linux-овое
- изначально в виндовс было микроядро (т.е. только необходимые ф-ции в ring0), а остальные были отдельными процессами в ring3
  это повлияло на то, что потом выделилась целая прослойка сервисов
- windows desktop ориентирован на массового пользователя, а значит сервисы должны быть такие, чтобы покрыть max к-во нужд
  без дополнтельных доустановок. потому сервисов довольно много и они живут в фоновых процессах

## Многозадачность
чтобы несколько процессов происходили "одновременно"
одновременность - иллюзия, т.к. при многозадачности CPU квантуется по времени - т.е. переключается между процессами очень быстро 
т.е., условно, встроенные часты тикают, тик1 - квант процессорного времени дан процессу1, тик2 - квант CPU времени дан процессу2, ...
кому дать CPU следующим, думает scheduler (это часть ОС).

процессы не просто стоят один за другим в очереди, у них есть приоритеты в зависимости от того, что они выполняют
(условно. оконный менеджер - высший приоритет, т.к. он постоянно обновляет окно 60р./сек., 
а синхронизация сообщейний whatsapp будет на фоне, более низкий приоритет)

очередей много, - столько же, сколько разных приоритетов у активных задач.
квантование времени CPU происходит м-у процессами в этих очередях по приоритетам. больше всего времени и чаще всего
будет отдаваться процессам с высшими приоритетами. внутри своей очереди у них равноправный круг. 
чтобы было справедливо, планировщик иногда будет давать CPU время тем задачам, которым давно его не давали.

многозадачность сочетается с мультитаскингом.
multitasking - ожидание естественной паузы процесса (окончия / ожидания ввода / ...) и только тогда передача CPU другому процессу
если процесс остановился, а его квант времени идёт, scheduler не будет ждать, когда квант закончится
он даcn CPU иному процессу
 
## Командная строка и графическая (оконная) среды взаимодействия с пользователем. 
Особенности.
CLI
- работает с потоком байтов из / в stdin, stdout, stderr - это абстрактные "файлы", которые даёт оболочка
- не принципиально, работать с файлом или выводить в консоль - cli не работает с выводом, это дело другого процесса

GUI
- работает с графикой и с сигналами от аппаратного обеспечения, нужна обработка событий 
  (типа мыши, т.к. её клик может означать "создать файл" и т.д.) 

Сравнение.
- из схожего: cli и gui - это графическая надстойка для обращения 
  к библиотеке сист. вызовов. выполняют одну и ту же функцию

- из различного: gui берёт на себя задачи, которыми не озабочен cli

Преимущества.
CLI
- больше возможностей. 
  например, из описанного в Таненбауме, в особенности конвейеры (sort ... | tail ...)
  я зачастю ипользовала GUI, и мне никогда не попадался такой инструмент в GUI
  это должно быть очень удобно, хотя бы потому что отсутствует запись с промежуточные файлы,
  чтение из них и т.д. 
- не занимается вопросами графики и взаимодействия с аппаратурой
  т.к. обязанности разделены, объект не перегружен.
  а когда у объекта слишком много функций, это значит, что его перегрузили
  а также ест меньше ресурсов 
- можно перенаправлять потоки in out err, также делается командой терминала
- bash / shell скрипты чтобы автоматизировать однотипные задачи

GUI
- более user friendly, упрощает взаимодействие с ОС и снижает порог входа во взаимодействие с ПК
- хорошо подходит для работы с визуальными задачами, т.к. GUI по своей сути визуально

Недостатки.
CLI
- высокий порог входа - нужно знать набор команд для взаимодействия с ОС
- по умолчанию, команды для удаления удаляют файлы безвозвратно 

GUI
- занимается и обработкой событий, и выводом, и вводом, и взаимодействием с библиотекой системных вызовов
- ест больше ресурсов
- ввиду особенностей назначения (ориент. на визуал), то, что в cli делается через одну команду (то же перенаправление <in >out)
  в gui будет кратно сложнее. потому что она не ориентирована на потоки данных, и ей надо будет это как-то обрабатвать
- для однотипных задач (типа пересохранить файл с другом формате) это будет множество ручных однотипных действий

## введение 
АО - аппаратное обеспечение, ПО - программное обеспечение
ПО использует АО, основывается на нём. ПО порождает процессы; 
процесс - программа во время выполнения. 
процесс - хранилище, в нём есть всё, необходимое для работы программы (оно лежит в адресном пространстве)
процессы взаимодействуют с CPU

CPU имеет 2 режима работы: режим ядра и режим пользователя
режим ядра - возможно исполнять всё, на что способно АО. 
режим пользователя - доступен ограниченный набор команд; например, нельзя читать / писать в произвольную память
syscall - когда процесс хочет сделать что-то ему недоступное, он делает syscall
          syscall переводит CPU в kernel mode
  	  происходит действие с АО
	  CPU переходит в user mode
	  процесс продолжается в user mode

ОС - это ПО, работающее в режиме ядра. Зачем она нужна:
общение "АО - пользователь" – это неподъемная задача
общение "АО - ОС - пользователь" – возможная задача, т. к. ОС вводит абстрактные ресурсы. через них: 
а) пользователь или ПО общается с АО; 
б) ОС и сама управляет этими абстрактными ресурсами
в) (от deepseek) ОС даёт изоляцию и защиту - процесс1 не касается памяти, которая используется процессом2

Пример на (а) 
"АО в виде жёсткого диска - драйвер - ОС - ПО". Так, ПО может создать "файл", 
хотя никакого "файла" не существует, это абстракция, которую дала ОС

Примеры на (б)
1. (разделение по времени)
какое-то ПО использует папку, а программист хочет её удалить. ОС не позволит ему это сделать,
т.к. эта папка используется другим процессом. 
2.(пространственное разделение - всем процессам по-немногу)
много процессов на одном ядре. ОС (планировщик) переключает их за очень малый промежуток времени
и все они используют одно ядро (ядро - вычислит. блок внутри CPU)

# процессы
когда происходит multitasking, то процесс принудительно останавливается. его состояние нужно где-то хранить
его тело лежит в его же *адресном пространстве (образе памяти)*,
а информация о том, на каком моменте он был остановлен,
(и все необходимые для возобновления данные) находятся в *таблице процессов*. 
это условно массив в котором лежат все активные процессы

*процесс может порождать дочерние процессы*

*межпроцессорное взаимодействие* - процессы могут быть порождены от одного процесса
тогда они выполняют смежные задачи. и им нужно обмениваться данными, а не быть полностью изолированными

*RAM - random access memory оперативная память устройства*
первая графа - то сколько RAM ОС дало процессу.
никогда не больше некого лимита
(либо ОС пересчитывает, чтобы было и на др. процессы;
либо явно определённый лимит. чаще первое)
 
*CPU - central processing unit (центральный) процессор*
вторая графа - сколько процесс съел CPU времени
процессы потребляют CPU время, когда они активны. в ином случае они "висят"
например, есть оконный менеджер dwm, который постоянно активен, т.к. он постоянно перерисовывет окно

процессов много. кол-во запущенных приложений < кол-во процессов

*КРАТКО: процесс попросил память; ОС дала адресное пространство (отображений нет);*
*процесс начал писать (page fault - выделение произошло, отображение сделалось);*
*процесс попросил то, чего нет в RAM - page out в pagefile чего-то из RAM, и page in нужного из pagefile в RAM*

РАЗВЁРНУТО
ОС не всегда всё держит в RAM. она регулярно использует C:\pagefile.sys (дисковое пространство), 
туда кладутся редко используемые вещи (page out), даже если в RAM есть свободное место 
если обратиться к тому что в pagefile, надо принести это в RAM (page in)
и возможно будет выполжено что-то из RAM (если там нет свободного места).
если такого "перекладывания" будет много, то это вызовет зависание

при обращении к тому, чего нет в RAM (есть в pagefile) будет аппаратное прерывание (pagefault). 
займёт милисекунды и потом процесс увидит обновлённый RAM
в который уже положено то, к чему он обратился

у каждого процесса есть адресное пространство, в рамках которого он может что-то делать
адресное пространство - "список" адресов, которые ОС позволило занимать процессу.
если процесс ничего ещё не писал, то не произошло отображения адреса из этой "таблицы" на физич. память
адресное пространство != реальный адрес. по адресу в АП может не быть привязки к pagefile / RAM
 
virtual memory - это механизм оторбажения адресов из адресного пространства на RAM и pagefile 
(никакому процессу это знать не надо)
MMU - memory management unit - это АО которое обеспечивает механизм virtual memory
MMU находится внутри CPU

сколько памяти сброшено в pagefile.sys: (Get-CimInstance Win32_PageFileUsage).CurrentUsage

2. multitasking VS multiprogramming

multitasking: на одном ядре процессы выполняются по очереди, но очень быстро
каждый процесс выполняется квант времени, потом scheduler думает какому процессу 
дать очередь следующему. процессы прерываются насильно 

multiprogramming: процесс1 выполняется на ядре, а когда он перестаёт выполняться и чего-то ожидает
то ядро отдаётся процессу2. процессы не прерываются насильно

одно не исключает другое;
если процесс начал зашёл в ожидание до того, как квант времени истёк,
то scheduler не будет жать окончания кванта, а отдаст управление другому процессу

3. time sharing (вариант многозадачности) (типа у каждого свой CPU)

несколько пользователей используют один CPU
пока user1 ничего не вычисляет, CPU отдаётся в пользование user2
чтобы CPU простаивал по минимуму

если при этом несколько юзеров одновременно хотят CPU 
то он будет квантоваться и даваться всем по кругу

так, может быть и 1 юзер. просто CPU будет распределяться м-у его процессами
а не м-у процессами многих юзеров

4. пакетная обработка VS интекрактивная программа
пакетная обработка - batch processing - это "дать инструкцию - взять результат"
ввода промежуточных значений нет, только Input Output
.bat файл то и делает - исполняет инструкцию

интекрактивная программа - ждёт когда юзер ей что-то введёт

5. некоторые процессы идут в фоновом режиме, а некоторые - в приоритетном

чем более приоритетен процесс, тем больше ему даётся квантов
и тем чаще scheduler отдаёт ему CPU

6. очередь процессов не одна, их много

сколько разных приоритетов текущих процессов, столько и очередей
например, есть приоритеты А Б С (от высокого к низкому) 
сначала scheduler будет выдавать кванты процессам с приоритетом А
потом процессам с приор. Б и так далее
но если какой-то процесс с Б долго простаивал, то ему scheduler
может дать кванты чтобы поддерживать справедливость

7. RAM подразделяется на уровни

CPU регистры - текущая операця
кэш - недавно использованные вещи
общее пространство - все активные процессы

ну и pagefile, но это уже не RAM, это диск(ssd/hdd см ssd-hdd-compare.png)

8. прерывания

*аппаратное прерывание* - когда от АО приходит сигнал что что-то произошло
информация об аппаратном прерывании может прийти в процесс, если это необходимо
(что-то было введено с клавиатуры или процесс вызвал ошибку)
int 0x... - это оно и есть. номер в таблице номеров прерываний
(int = INTerrupt)

*программное прерывание* - процесс делает syscall к CPU, тот переходит в режим ядра и что-то выполняет
работает не через int 

в момент любого прерывания текущий процесс ставится на паузу
процесс с kernel mode может запретить прерывать его аппаратно
процесс с user mode не может запрещать прерывать себя

у обоих категорий прерываний есть приоритеты, как и у процессов

9. системные вызовы

процесс не делает syscall-ы напрямую. он вызывает нужные библиотечные функции ОС, которые уже внтури делают реальные syscall-ы

10. core VS kernel 

core - ядро в CPU, физический модуль
kernel - ядро ОС. отвечает за управление процессами, scheduler, работа с памятью, 
	 и пр. задачи связанные с непосредственным взаимодействием с аппаратным обеспечением

11. монолитное ядро и микроядро

в монолитном ядре многие функции взаимодействия с АО живут в этом ядре. и работают в kernel mode

в микроядре живут только самые важные функции. а остальные являются отдельными процессами, "сервисами",
  которые уже в user mode. и вот когда надо например прочиатать что-то, то будет цепочка
  процесс -> микроядро[3-0] -> файловый сервер -> (чтение диска, syscall) микроядро [3-0] -> файловый сервер [0-3] -> процесс[0-3]

у виндоус модифицированное микроядро (по факту монолитное)

микроядро даёт замедление по времени и безопасность, т.к. если упадёт процесс с ring 3, то это не положит микроядро 
а вот если упадёт что-то внутри монолитного ядра, то оно положит ядро
