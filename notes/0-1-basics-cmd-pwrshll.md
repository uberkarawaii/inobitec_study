## Командные оболочки Windows: PowerShell и cmd

Современная основная оболочка разработчика на Windows — 
**PowerShell** (cmdlet'ы по схеме `Verb-Noun`, объектная модель пайплайна вместо текстового, 
полноценный скриптовый язык, кросс-платформенность через PowerShell 7). Классический `cmd` остаётся для 
совместимости со старыми сценариями `.bat` и инструментами, которые от него зависят.
Понимание разницы между объектным пайплайном (передаёт типизированные объекты) 
и текстовым пайплайном (передаёт строки), профили (`$PROFILE`), execution policy.

### Power Shell

- оболочка командой строки

- возможность писать исполняемые инструкции

- платформа для управления конфигурацией.
  конфигурация - это состояние компа. что установлено, какие права у пользователя, и т. д.
  администратор не будет руками настраивать все компы. 
  легче написать один скрипт и выполнить его для всех компов, тогда они будут в нужном состоянии
  платформа позволяет сделать описанное выше, а также следить и поддерживать состояние платформы

Power Shell принимает и отдаёт объекты .NET, в не текст. 
и когда делается цепочка команд через | то результат операции передаётся в след. 
операцию не как текст, а как объект .NET

в Power Shell тоже можно делать cd, mkdir, ... Linux-style команды
но они - лишь псевдонимы для реальных команд Power Shell с таким же значением; для проверки псевдонимов: Get-Alias ...
стиль команд Power Shell: Verb-Noun. пишется в CamelCase хотя сам PowerShell не чувствителен к регистру

т.к. работа тут именно с объектами, то можно сделать не только:
``` 
PS C:\Users\User> Get-ChildItem | Select-Object -Last 1 | Select-Object -Property Name

Name
----
urllib3-1.26.7-py2.py3-none-any.whl
```

но и так:
```
PS C:\Users\User> (Get-ChildItem | Select-Object -Last 1).Name
urllib3-1.26.7-py2.py3-none-any.whl
```

в первом случае это отфильтрованный объект, у которого оставлено 1 свойство
во втором случае выводится значение поля 
т.е. при выводе "табличек" Power Shell показывает именно объекты

#### Кросплатформенность Powershell 7

Powershell 5.1 и ранее - .Net Framework; с Powershell Core (6) - .Net 

powershell 7 построен на .Net (Core/8), который в отличии от .Net Framework не привязан к Windows. 
.Net-механизм похож на Java-механизм:
есть своя среда исполнения под каждую ОС (.Net Runtime), 
в которой может исполняться .dll/.exe, внутри которого - IL (intermediate language).
intermediate, т.к. он между ЯП и машинным кодом. в рантайме IL становится машинным кодом

.Net Runtime отвечает за исполнение в разных средах, а исходник (в данном случае - приложение .exe) 
может быть переносимв любую ОС без дополнений / исправлений

в Powershell на Windows будет доступно больше команд чем на Linux/macOS.
(*информация от deepseek*) тот же Execution policy который у меня был доступен только под виндой

### cmd - зачем нужен

- не все вещи нуждаются в сложностях, которые умеет делать Powershell. для простых команд достаточно текстового пайплайна

- powershell тянет за собой .net runtime, который довольно тяжёлый. а cmd мало весит и умеет исполнять простые вещи.
  как раз для них cmd подходящ и по этой причине

- уже есть вещи, которые работают через cmd, и переписывать их под Powershell нет смысла

### объектный и текстовый пайплайн

когда в текстовом выполняется что-то такое
```
C:\Users\User\Desktop\inobitec_stud>dir | findstr .md
14.06.2026  09:12            30 523 00-temy.md
14.06.2026  09:12            18 699 01-teoriya.md
...
```
то по сути dir выдаёт текст, findstr выполняет поиск по тексту регуляркой. findstr не обращается к "полю" .md или 
как-то так, он именно ищет указаную строку в тексте. и передаётся между командами именно текст 

при выполнении в powershell
```
PS C:\Users\User\Desktop\inobitec_stud> Get-ChildItem | Where-Object { $_.Extension -eq '.md'}


    Каталог: C:\Users\User\Desktop\inobitec_stud
Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
-a----        14.06.2026      9:12          30523 00-temy.md
-a----        14.06.2026      9:12          18699 01-teoriya.md
-a----        14.06.2026      9:12         224811 02-praktika.md
-a----        14.06.2026      9:12          14908 03-ocenka.md
-a----        21.07.2026     12:00         111186 AGENTS.md
-a----        06.07.2026     13:51           5379 README.md
```

Get-ChildItem выдаст объекты (System.IO.FileInfo) и передаст в Where-Object именно объекты, не текст. Where-Object будет
обращаться к полю .Extension у всех объектов, а не просто делать поиск регуляркой 

то есть, текстовый пайплайн: передаёт м-у командами текст, внутри команды работает с текстом

объектный пайплан: м-у командами передаются объекты, внутри команд работает с объектами 

### профили powershell 

это некая конфигурация среды, которая делает каждый сеанс персонализированным: добавление команд, переменных, 
псевдонимов,
и т.д. при этом, профиль - это .ps1 файл, который запускается при запуске powershell. пользователь
должен сам создать файл профиля, т.к. powershell не делает профилей сам. по дефолту путь до профиля пустой.

есть 4 уровня профлией:

1. Все пользователи, все хосты
2. Все пользователи, текущий хост
3. Текущий пользователь, все хосты
4. Текущий пользователь, текущий хост

**хост** - программа, использующая powershell-движок. и все эти "политики" будут применяться сверху вниз (1 -> 4).
и если есть повторения, то, что есть на "текущ. пользователь, текущ. хост" будет переопределять всё что было выше
(если на это нет запрета от администратора в AllUsersAllHosts; и запреты могут быть на изменения переменных /дисков, 
а не на всё подряд). 

по переменной $PROFILE можно перейти к путям профилей, доступных из текущего сеанса. например:

- $PROFILE = $PROFILE.CurrentUserCurrentHost (текущ. пользователь, текущ. узел)
- $PROFILE.CurrentUserAllHosts (текущ пользователь, все узлы) 
- и т.д.

### execution policy

политика, отвечающая за исполнение скриптов .ps1; 
на виндоус хранится в реестре. применяется при загрузке powershell,
до загрузки профилей. профили не могут обойти execution policy.

существуют следующие политики:

- Restricted — .ps1 не запустится, пока ты не сменишь политику
- RemoteSigned — локальные .ps1 запускаются, скачанные из интернета — нет (нет подписи)
- AllSigned — только подписанные скрипты (любые)
- Bypass — ничего не блокируется, ничего не предупреждается 

exec. policy - не система безопасности. это блокировка от непреднамеренных нежелательных событий. если пользователь хочет
совершить какое-то действие, то он может либо прописать тот же скрипт в консоли, либо сменить exec. policy через реестр
