## 2. ПРАКТИКА

Найти приемлемый инструментарий и попрактиковаться. В зависимости от предпочтений и того, что найдёте исходя из изученного в [§1](01-teoriya.md) — скачайте необходимое программное обеспечение и, следуя указаниям ИИ, попробуйте самостоятельно написать простейшую программу на предпочтительном языке программирования.

Здесь вам необходимо будет выбрать произвольную задачу. Источники:

- **Современные платформы с разбором решений:** [Codeforces](https://codeforces.com), [LeetCode](https://leetcode.com), [Kattis](https://open.kattis.com), [HackerRank](https://www.hackerrank.com), [Codewars](https://www.codewars.com), [Project Euler](https://projecteuler.net), [Advent of Code](https://adventofcode.com).
- **Классические задачники:** Златопольский («Сборник задач по программированию»), Шень («Программирование: теоремы и задачи»), Окулов («Программирование в алгоритмах»).
- **ИИ-генератор:** запрос к ChatGPT / Claude / DeepSeek с указанием уровня и языка.

Хорошая привычка — одна задача в день: даже простая задача с написанным тестом даёт больше, чем чтение учебника без практики.

Важно активно самостоятельно практиковаться, написать программу, запустить её и получить какой-то результат. Использовать ИИ в качестве помощника в изучении — при возникновении проблем с каким-либо этапом работ.

Используйте материалы и лучшие практики признанных экспертов в области программирования, например <https://github.com/cpp-best-practices/cppbestpractices>.

### 2.1. Минимальный обязательный спектр навыков (C / C++ трек)

#### 2.1.0. Методические принципы освоения трека

Таблица навыков ниже — это не план «пройти по строкам сверху вниз», а **карта слоёв**, которыми обрастают учебные задачи. Курс устроен по трём принципам.

**1. Сначала вручную, потом автоматизация.** Прежде чем подключать `make` — собрать программу руками вызовом `cl.exe` / `g++` / `clang++`. Прежде чем подключать `cmake` — написать Makefile. Прежде чем подключать тестовый фреймворк — написать тест как обычный exe и убедиться, что он возвращает осмысленный код возврата. **При первой встрече с внешней библиотекой** — собрать её один раз из исходников (`./configure && make && make install` или `cmake .. && cmake --build . && cmake --install .`), чтобы увидеть, что vcpkg/apt автоматизируют. Каждый верхний слой имеет смысл только тогда, когда понятен слой под ним; иначе он превращается в «магию», которую студент использует, не понимая, что она делает. Метасборщик без понимания сборщика вреден.

**Уточнение про vcpkg.** Принцип «сначала вручную» — это **тренировочный шаг при первой встрече** с конкретной библиотекой (один раз собрал `gtest` руками, понял, что внутри, — дальше используешь vcpkg). Это **не позиция vcpkg в лестнице** — сам инструмент vcpkg подключается в естественном месте, как только появилась первая внешняя зависимость (между метасборщиком и тестовым фреймворком), не «в самом конце курса».

**2. Задачи живут в общей инфраструктурной рамке.** На любом этапе у студента не один проект, а несколько небольших задач (например, разные задачи из задачника, мини-CLI, упражнения по теме недели), и **все они одновременно растут под общей рамкой**: один шаблон CMake, один логгер, один формат лога, общий стиль кода, общая сборка. Когда добавляется новый слой (`vcpkg`, тестовый фреймворк, статический анализатор, packaging) — он подключается ко всем задачам сразу, а не к одной. Это и есть то, что в индустрии называется «проект»: задач много, инфраструктура общая.

**3. Инструменты — рамка задачи, а не предисловие.** Стартовая инфраструктура (файловый менеджер, командная строка, компилятор, git, минимальный CMake-шаблон, `clang-format`, статический анализатор) ставится в первую неделю — и **первая же задача по программированию решается уже в этой рамке**. Не «месяц инструментов перед задачами», а «задача с первого дня + инструменты подключаются по мере того, как задача требует нового слоя».

**4. Темп подстраивается под уровень студента, не наоборот.** Лестница слоёв в таблице § 2.1.A (окружение → ручная компиляция → раздельные этапы сборки → пакетный сценарий → сборщик → метасборщик → пакетный менеджер) — это **диагностика**, а не план гонки. Если студент уверенно владеет PATH/INCLUDE/LIB, ключами компилятора, разницей `cl.exe` и `link.exe` — двигаемся быстро. Если на каждом слое нужно учить с нуля — это пробел, который **не пропускается ради темпа**, а закрывается на этом же слое. Иначе верхние слои становятся карго-культом: студент использует CMake, не понимая, что тот генерирует Makefile, который вызывает компилятор и линковщик.

**Уточнение (2026-06): пробел слоя 0 ортогонален уровню кода.** Кандидат может писать код уверенного junior-плюс / middle-уровня (алгоритмы, оптимизации, продуманные структуры) — и при этом собирать всё исключительно кнопкой IDE, без единой CLI-сборки, make или CMake в каком-либо из своих проектов (наблюдение на кандидате трека C 2026-06: четыре доведённых до продукта репозитория, ноль команд сборки; исторически — тот же пробел у выпускницы, утонувшей в ошибках линковки на реальном проекте). Высокий языковой уровень **не предсказывает** владение фундаментом сборки — диагностировать слой 0 отдельно, не выводить из качества кода.

**Уточнение (2026-06): карго-культ слоя возможен и без перепрыгивания.** Пропуск слоёв — не единственный его источник: студент может пройти лестницу строго по порядку и всё равно подключить слой механически. Наблюдение на кандидате трека C: `make` взят после `build.sh`, PCH — после раздельной сборки (порядок соблюдён), но граф зависимостей make не используется для данных и тестов (они остались в shell-сценариях), прекомпилированный заголовок собран копированием инклюдов из файлов (противоположно смыслу PCH), а сломавшееся паттерн-правило залатано явным правилом со скопированной бессмысленной зависимостью. Исторически то же у выпускника, дошедшего до CMake. **Факт наличия слоя в репозитории не диагностичен** — диагностика идёт по артефактам слоя и практическим вариациям: «объясни каждую строку `.d`-файла», «что произойдёт при изменении pch», «перенеси тест-кейс в граф make» (active recall по принципу 5), не «слой подключён — галочка».

**Дополнительный порядковый принцип — инфраструктура тестирования полностью построена до оптимизаций.** Сначала: юнит-тест (без фреймворка → GoogleTest) + acceptance (через перенаправление+diff → генератор → матричный suite с `boost::process`) + покрытие — всё собранное и работающее. **Потом** — алгоритмическая адаптация, параллелизм, PGO. Меняя алгоритм или подключая параллелизм, прогоняем матричный suite → видим, не сломалось ли поведение программы → меряем `std::chrono` → решаем оставлять. Без полной testbed оптимизация = слепая работа, регрессию заметить трудно. На уровне таблицы § 2.1.A это означает: строки про тестирование («Тестирование», «End-to-end тестирование CLI», «Метрики покрытия кода») идут **до** строки «Оценка алгоритмической сложности и адаптация под ресурсы» и далее. Аналогично — **локализация после vcpkg/CMake**: `gettext`/`boost::locale` требуют пакетного менеджера и заметно нагружают сборку, без PCH становятся болью; на учебной программе локализация — слой UX-полировки, подключается ближе к финалу разработки, не на 1-й неделе.

В оценочных периодах (например, при подготовке к стажировке) скорость прохождения по лестнице — главный сигнал об уровне кандидата. «За месяц дошёл до метасборщика и пакетного менеджера» и «за месяц дошёл до сборщика с пониманием потока зависимостей» — это два разных уровня готовности; оба валидные результаты, но первый ближе к продуктивной работе, второму ещё нужно учиться.

**5. Работа с AI-помощником: усвоение как узкое место.** AI-инструменты разработки (Claude Code, OpenCode + DeepSeek, Cursor, Aider — см. § 0.2) резко сокращают время на производство кода: рабочий вариант учебной задачи студент получает за часы. Это сдвигает узкое место с **производства** на **усвоение**, и контрольная точка курса — **работа без AI на чистой ВМ под наблюдением наставника**. Студент поднимает окружение с нуля, повторяет ключевой кусок задачи и проходит небольшие изменения постановки в моменте. Это самый чистый сигнал, что период дал усвоение, а не пассажирство при AI; косвенные диагностики (опросы про устройство инструмента, тесты на распознавание галлюцинаций, задачи на параллельную постановку, которую студент не делал) подменяют прямой сигнал и менее надёжны.

Целевой уровень владения инструментом — **инженерный, не внутренностный**: знаю **когда** применять, узнаю **когда не работает**. Если студент включил PGO на CPU-bound коде, увидел отсутствие прироста и предположил «training data нерепрезентативно или код IO-bound» — этого достаточно. Знание устройства самой PGO-инструментации (как и устройства GoogleTest, vcpkg, статанализаторов) — отдельный путь дипломного R&D. *Пассажир* применяет, не зная когда; *автор* знает внутренности; курс прицеливается в *инженера* посередине.

Изменения постановки на контрольной — **вариации в рамках текущей задачи студента**, не параллельные постановки. Причина: студент держит в голове модель того, что **делал**, и расширение этой модели измеряет глубину усвоения; параллельная задача требует переключения и измеряет уже другое.

Сигнал про работу с AI живёт в **одном артефакте — сырых машинных логах AI-сессий** (`dialog_logs/*.json` — машинный экспорт через штатную команду инструмента, не ручная транскрипция). Полный экспорт переписки целиком, без редактуры/выжимок/рефлексивных конспектов пост-фактум — обработанный конспект и ручная транскрипция студента одинаково плохи: оба выкидывают именно те детали (tool-calls с аргументами и выводом, точные формулировки, временные метки), по которым видна манера работы с AI. На сессии без AI нельзя увидеть, как студент с AI взаимодействует — сырые логи единственный канал. Грейдятся наставником по трём уровням: *пассажир* («AI написал, я применил»), *критический пользователь* («AI предложил X, я заметил, что не работает потому что Y, переспросил так-то»), *соавтор* (видна собственная инженерная позиция, AI — инструмент). Это не дополнение к коду, а отдельный сигнал, по весу сопоставимый с самим кодом. Конкретные команды выгрузки для каждой обвязки (OpenCode, Claude Code, прочее) — в [03-ocenka.md](03-ocenka.md) § «Защита от AI-плагиата» пункт 1.

**Формат сырых логов — инструментальная задача наставника на онбординге, не дисциплинарное требование к студенту.** Студенты, услышав «нужны сырые логи», по умолчанию интерпретируют это как «нужно написать подробнее своими словами» — ручная транскрипция, табличные конспекты с рефлексией, попытки «правильно описать» диалог. Это не непослушание: формулировка «сырой лог» техническая, и без показа, **где именно** AI-инструмент хранит сессии и какой командой их экспортировать, студент дрейфует к выжимкам. На первой совместной сессии наставник демонстрирует команду экспорта (`opencode export <id> > dialog_logs/<тема>.json` или аналог), открывает получившийся JSON, показывает структуру (роли, tool-calls, временные метки) — после этого требование «сырой лог» становится конкретным. Без этого шага неделями идут уточнения «нет, не пересказы, а исходники», и накапливается несоответствие между ожиданием наставника и работой студента.

**Каналы проверки усвоения: опросные формы фальсифицируемы ИИ (уточнение 2026-06).** Письменный ответ на вопросы понимания («ответь по пунктам в письме») студент может сгенерировать тем же AI-ассистентом; устный опрос на встрече — подготовить и сопровождать им же. Достоверен только практический канал: чистая ВМ, руки на клавиатуре, вариации в моменте. Следствие для учебных материалов: вопросы на понимание в студенческих документах формулируются как **самопроверка** («разбери для себя», «сделай и посмотри, что изменилось»), не как отчётность; наставник — не экзаменатор, понимание видно косвенно, через практические вариации. Зафиксированный анти-паттерн наставника (включая агента-наставника): тяга оформить проверку понимания письменным или устным отчётом — дешевле в моменте, но канал скомпрометирован.

**Self-awareness пассажира — не сигнал перспективности (уточнение 2026-06).** Студент-пассажир может трезво осознавать свой профиль («фактически я написал один промпт») и спокойно соглашаться с диагнозом, не торгуясь за оценку. Это не готовность к пересмотру, а отстранённость: цель — зачёт/корочка, не компетенция. Три уровня этого принципа (пассажир / критический пользователь / соавтор) описывают **компетенцию**, не мотивацию; осознанный пассажир остаётся пассажиром. Эталонный внешний случай (2026-05): курсовая сдана одним промптом агенту, студент открыто признал и согласился с оценкой «по сути ноль» — и подход не изменил. Формула отказа для таких ситуаций: «это то же самое, что дать опытному разработчику сделать работу за тебя; цель обучения — чтобы ты смог без ИИ; посади на чистую виртуалку без ИИ — ничего не сделаешь».

**6. Голая версия с первого дня, промышленная позже (спираль освоения).** Каждый инфраструктурный инструмент (тестирование, логирование, версионирование, форматирование, e2e-тесты, статанализ) имеет **учебный голый аналог** — из стандартной библиотеки или базового toolchain. Голая версия вводится **с первой задачи**. Промышленная (фреймворк или сторонняя библиотека через пакетный менеджер) подключается **позже**, когда задача дорастает: голая упёрлась в видимое студенту ограничение, промышленная даёт конкретную ценность.

Принцип отличается от принципа 1 («Сначала вручную, потом автоматизация»): принцип 1 — про **глубину** освоения одного слоя (компилятор → make → CMake — каждый раз понимаем, что автоматизирует следующий слой); принцип 6 — про **широту** покрытия инструментов с первого дня (тесты, логи, версии, формат — всё в простейшем виде с первой задачи, а не «вот сначала освою vcpkg, потом GoogleTest, потом начну писать тесты»).

Почему: привычка пользоваться инструментом формируется в процессе работы с инструментом, не до него. Если ждать промышленных версий — студент пишет код без тестов / без логов / без CHANGELOG несколько недель, потом ставит инструменты, но уже без привычки их использовать. Обратное направление дёшево: голая версия в первый день, промышленная — естественный переход.

Примеры пар для C/C++ трека (полная таблица — в персональных `AGENTS.md` соискателей):
- **Юнит-тесты функций:** exe с `return 0/1` (отдельный `main`, вызывает функцию, проверяет результат) → GoogleTest + ctest.
- **E2E-тесты CLI:** acceptance через перенаправление и `diff` (`./prog < test_data.txt > out.txt && diff out.txt etalon.txt`, файлы руками) → расширение с генератором детерминированного входа (`std::mt19937` фиксированный seed) для массовых входов → матричный acceptance с `boost::process`.
- **Логирование:** `std::cerr << "msg\n"` (без обёртки, без уровней, без timestamps) → промежуточные ступени по нужде (уровни → timestamps → метка модуля) → `spdlog`.
- **Версионирование:** `kVersion` в коде + `--version` + `CHANGELOG.md` → CMake `project(VERSION)` + `git describe` + `SOVERSION` (для библиотек).
- **Форматирование:** `.clang-format` в репо с первого коммита → pre-commit hook + CI-проверка.
- **Статанализ:** `-Wall -Wextra -Wpedantic` → `-fanalyzer` → внешние `clang-tidy`/`cppcheck`.

Для Python / JS-TS / Java / других треков пары структурно те же: ручной `assert` + `pytest` без покрытия → `pytest` + coverage + параметризованные; `print` для логов → `logging.handlers` / `winston` / SLF4J; `version.txt` → семантическое версионирование в `pyproject.toml`/`package.json`/`pom.xml`. Конкретные пары определяются стеком.

На ревью наставник смотрит: есть ли голая версия инструмента уже на 1-й неделе. Если её нет (тестов нет ни в каком виде, нет CHANGELOG, нет формат-конфига в репо) — это сигнал, что привычка не закладывается; промышленная версия на 2–3 неделе **не компенсирует** отсутствие голой на 1-й.

**При переходе на промышленную** между несколькими кандидатами (например, для логирования — `spdlog` vs `Boost.Log` vs `log4cpp`) — выбирать по чек-листу в § 2.1.E «Критерии выбора промышленной библиотеки». Без чек-листа выбор сводится к «AI-помощник посоветовал» или «в учебнике использовали» — оба слабые сигналы.

**7. `AGENTS.md` как runtime config, не справочник.** Правило, прописанное в `AGENTS.md`, **не применяется автоматически** — даже когда студент его видел и формально подтвердил. Декларативный формат текста в `AGENTS.md` не активен в моменте действия; активна процедура, привязанная к моменту действия. Для каждого правила должна быть **процедура применения в моменте**:
- Строка в **стартовом промпте AI-сессии** (анти-подхалимажные приёмы § 4.2 → AI сам инициирует их каждую беседу, без напоминания студенту).
- **Hook в `.claude/settings.json` или pre-commit** (например, `grep -nE 'Ошибка|Error|Failed|Не удалось'` по diff'у с подсветкой шаблонных формулировок § 4.6; `clang-format --dry-run -Werror` автоматический, не «руками перед коммитом»).
- **Формулировка правила как «триггер → действие»**, не «правило → применять при случае».

Без процедуры `AGENTS.md` остаётся справочной литературой: на ревью наставник видит, что правила прописаны, в коде не применены. Это **не нарушение со стороны студента**, это структурное свойство декларативного формата. Каждое новое правило в `AGENTS.md` сопровождается процедурой; если процедуры нет — правило не активно.

**8. Анти-подхалимажный приём — процедура с финальной проверкой, не только в старте.** Приёмы § 4.2 студенческого `AGENTS.md` («не верь утверждениям», «оцени уверенность 1–10», «факт/мнение/предположение») применяются на **старте** ревизионной серии — формулировками типа «будь строгим руководителем» в первой реплике сессии. **В финале серии режим выключается:** когда AI выдаёт «10/10, претензий ноль» или «огромный прогресс, это идиоматический C++», студент принимает оценку без вызова. Это **классический подхалимаж модели** — AI оптимизирован выдавать удобный ответ к моменту закрытия задачи, и без явной провокации опускает планку.

Решение — явная **финальная анти-подхалимажная формула**: «по итогу каждой задачи, прежде чем закрыть сессию: оцени уверенность в этом ревью 1–10; если не 10 — что ещё могло остаться?». Формула включается **в стартовый промпт AI-сессии** (см. принцип 7) — тогда AI сам инициирует финальную проверку, не зависит от того, что студент вспомнит её применить. Закрывает дыру финала одной строкой в шаблоне промпта.

**Эмпирическая проверка процедурной активации (2026-06, 1 студент, положительная).** У первого студента, которому финальная формула и требования § 4.2 были встроены в стартовый промпт (активация 2026-05-23), в следующей же итерации паттерн «финал без вызова» не воспроизвёлся: после каждого «исправил» студент сам инициировал дополнительные раунды («найди ещё», «проверь максимально строго», «найди несостыковки с review») — до сухого остатка, по всем задачам недели. Накапливаем подтверждения на следующих студентах.

**Связанный паттерн — выборочное применение рекомендаций AI без явной маркировки пропуска.** AI выдаёт чек-лист из N пунктов (например, переписать код по 8 идиоматическим правилам); студент применяет часть, оставшиеся **молча игнорирует**, без отметки «пункт K не сделал потому что Y». AI на повторных ревью **не возвращается к пропущенным пунктам** — это та же сторона подхалимажа (модель опускает планку, чтобы продолжить хвалить). Финальная формула выше частично закрывает: AI на финальной проверке вспоминает свой первый план и видит расхождение. Дополнительно — в стартовом промпте требование к студенту: «при отказе от рекомендации AI — явная отметка в диалоге, "пункт K не делаю, причина Z"». Это и обучает явному обоснованию решений, и оставляет AI повод вернуться, если причина слабая.

**9. Документация инструмента читается раньше вопросов к AI.** Типовой студенческий промах — задавать AI базовые вопросы про команды инструментов: «как закомитить», «как экспортировать сессию», «как выйти из режима просмотра git diff». Ответы находятся в `--help` / `man <tool>` / README **первым шагом**. AI даёт правильный ответ, но **подменяет навык чтения документации** — отдельную компетенцию, без которой на новом инструменте студент тонет в production.

Правило для `AGENTS.md` (расширение § 4.2):
- Перед вопросом к AI про команду инструмента — открыть `--help` / `man <tool>` / README первой главы.
- AI применяется когда документация неоднозначна, нужна интерпретация, сравнение нескольких подходов, или речь не про команду инструмента (идиома языка / архитектурный выбор / отладка).
- При обращении к AI про инструмент — **давать цитату из документации**, что именно непонятно. Это одновременно проверка модели против источника (анти-галлюцинационный приём § 4.2).

**10. Обработка ошибок — сквозная дисциплина, не локальное упражнение.** Обработка ошибок в C/C++ — структурный пробел индустрии: программистов учат «happy path», ошибочные ветки приходят как поздние поправки через bug reports. Причины структурные, не «программисты ленивые»: вузовские курсы фокусируются на алгоритмах, не на полном контракте функции; C++ предлагает 5+ моделей одновременно (коды возврата, `errno`, исключения, `std::optional`, `std::expected<T, E>`, sentinel-значения) без единого паттерна; тестирование ошибочных веток дороже happy-path (нужна fault injection); code review редко придирается к error path; production-классы ошибок (timeout cascades, OOM-killer, partial failures, signal handling) в учебных программах не появляются. Курс развивает обработку ошибок **сквозной темой через все 4 недели**, не оставляет её локальной задачей одной недели.

**Архитектурный выбор для C++-стека курса — `std::expected<T, E>` основная модель, исключения ознакомительно.** Тренд стандарта C++23 (явно ввёл `std::expected` как первоклассный механизм типизированных кодов возврата), production-практика крупных проектов (Google C++, LLVM/Clang, WebKit, Chromium, Unreal Engine собираются с `-fno-exceptions`), специфика системного / multi-core / embedded / HFT стека (multi-core stack unwinding имеет фундаментальные проблемы с глобальным мьютексом; ABI-границы с C-API типа OpenGL/Vulkan/CUDA не пропускают исключения; real-time код требует детерминированной latency). Это **архитектурное решение под целевой стек курса**, не догма: для прикладного C++ (Qt-приложения, бизнес-серверы с low-frequency requests) исключения остаются нормальной моделью.

**Что значит «ознакомительно» про исключения.** Студент **знает механизм** (поведение под `throw`, exception safety guarantees basic/strong/nothrow, RAII в деструкторах, `noexcept` спецификация), **умеет читать legacy-код с исключениями** и оборачивать их в noexcept для C-ABI границ. **Применяет точечно** — конструкторы (нельзя вернуть `expected` из конструктора), stream-exceptions для IO-ошибок в декларативных pipeline (маска `badbit`, не `failbit`), интероп с библиотеками, бросающими исключения. **Не использует** как основную модель control flow ошибок прикладной логики.

**Лестница моделей по неделям курса** (детали — в § 2.1.E «Архитектура обработки ошибок»):

- **Неделя 1 — базовая корректность.** State-flags + коды возврата (`sysexits.h`: `EX_USAGE=64`, `EX_DATAERR=65`, `EX_NOINPUT=66`, `EX_IOERR=74`); классификация ошибок ввода (EOF / fail / bad для C++; `errno` / `feof` / `ferror` для C); двухканальный путь exit + stderr (exit code = категория для shell-пайплайна, stderr message = детализация для человека через `errno` + `ec.message()`).
- **Неделя 2 — типизированные модели и исключения как механизм.** `std::expected<T, E>` (C++23) глубже, monadic operations (`and_then` / `or_else` / `transform`), тестирование ошибочных веток как обязательный класс тестов (fault injection для IO/malloc/network). Исключения — ознакомительно: механизм, exception safety guarantees, `noexcept`, обёртки для C-ABI. Stream-exceptions для IO в pipeline (`badbit`-маска, платформо-специфичность `SIGPIPE` через `#ifdef`).
- **Неделя 3 — локализация сообщений об ошибках.** ru/en по системной локали, формат `progname: file:line: severity: message` (GNU Coding Standards § 4.3 Errors); стиль сообщений по § 2.1.C (конкретика, не штамп).
- **Неделя 4 — production-обработка.** Постмортем (core dumps на Linux, minidump на Windows), split debug info (`objcopy --only-keep-debug` + `strip --strip-debug`), корреляция логов с crash-дампом по build-id, символ-сервер.

**Уровень оценки готовности к стажировке** (расширение § 3 «Критерии оценки» в студенческих `AGENTS.md` для треков C / B): студент **различает классы ошибок ввода** (EOF/fail/bad/errno), **использует sysexits** для exit-кодов с двухканальным разделением exit+stderr, **владеет `std::expected<T, E>`** как первой моделью обработки ошибок прикладной логики, **знает exception safety guarantees** ознакомительно и **умеет читать legacy-код** с исключениями. Это **редкая на собеседованиях компетенция**: программистов с алгоритмическим уклоном много, программистов с дисциплиной обработки ошибок — мало. Тема — отличительная черта курса.

**11. Эволюция работающего кода под изменение условий — отдельная компетенция.** Большая часть индустриальной работы инженера — не написание с нуля, а возврат к работающему коду через недели/месяцы под изменившиеся условия: вырос объём данных и производительность упала; новое требование клиента; смена API стороннего сервиса; нагрузка с low-traffic стала high-traffic; программа перестала работать на новых данных, которые в первой итерации не учитывались. Это **отдельная компетенция**, которая в вузовских курсах структурно не осваивается: учебная логика «написал → сдал → забыл» не создаёт ситуации возврата, а в индустрии возврат — норма.

В курсе компетенция формируется через **«вторую итерацию задачи»**: через 2–4 недели после первой сдачи наставник возвращает студента к одной из ранее закрытых задач с конкретным триггером — объём данных вырос в N раз, latency-требование ужесточилось, формат входа сменился, добавилось требование streaming-обработки вместо batch, профиль нагрузки сменился (раньше batch, теперь high-frequency). Студент рефакторит, не переписывает с нуля.

**Дисциплина «второй итерации»** (по индустриальной практике, основа — Michael Feathers «Working Effectively with Legacy Code», 2004):

1. **Читаем свой код как чужой.** Без свежей памяти о решении видна реальная читаемость: понятны ли имена, видна ли структура, есть ли doc-comments на публичных функциях. Это — **главный диагностический сигнал качества первой реализации**: если через 3 недели студент не может разобрать собственный код за 15 минут — первая итерация была написана для сдачи, не для жизни.
2. **Характеризуем тестами.** Если на первой итерации тестов было мало — добавляем **characterization tests** (фиксируем текущее поведение в тестах, чтобы поймать регрессию при рефакторинге). Тест ловит то, что код **делает**, не то, что **должен делать** по ТЗ — разница принципиальная: characterization-тест защищает от случайных изменений поведения, не верифицирует корректность.
3. **Профилируем** (если изменение про производительность). Измеряем **до** правки — какой именно участок узкое горлышко на новых условиях. Без замера = «оптимизируем на глаз», та же ловушка, что в строке 147 § 2.1.A.
4. **Минимальный рефакторинг под новое требование.** Меняем то, что мешает выполнить новое требование, не больше. Сохраняем структуру, где возможно. Это **противоположность** реакции «всё равно много работы — проще переписать».
5. **Сверка с замерами до/после.** Тесты проходят, метрики улучшились, регрессий нет.

**Антипаттерн «переписать с нуля»** (Joel Spolsky «Things You Should Never Do, Part I», 2000) — одна из самых дорогих ошибок в индустрии. Netscape rewrite, который дал Microsoft три года форы и фактически стоил Netscape рынка браузеров, — каноническая иллюстрация. На учебном уровне антипаттерн проявляется как «у меня нет времени читать свой старый код, проще написать снова» — это **сигнал**, что код первой итерации был непригоден для возврата (нечитаем, не покрыт тестами). Правильная реакция наставника — вернуться к **первой итерации** и исправить её (читаемость + тесты), не разрешать переписывать заново.

**Связь с другими принципами:**
- **Принцип 5 (AI)** — на «второй итерации» работа с AI особенно показательна: *пассажир* теряется на чужом (своём же) коде, не понимает, что писал AI; *критический пользователь* читает и точечно правит; *соавтор* пересматривает структуру под новое требование.
- **Принцип 6 (голая версия с первого дня)** — голые тесты, лог, формат-конфиг на первой итерации = обязательное условие, чтобы «вторая итерация» вообще была возможна. Без тестов на первой — невозможно характеризовать поведение и поймать регрессию.
- **Принцип 12 «Локальная правка вместо архитектурного пересмотра»** — на «второй итерации» проявляется особенно ясно: локалист патчит точечно там, где новое требование требует архитектурного пересмотра, и упирается в потолок; архитект — пересматривает структуру. Это **главный момент**, когда наставник видит склонность.

**Диагностические вопросы на встрече по «второй итерации»:**
- Где в твоём коде структура помешала тебе добавить новое требование? Что бы ты сделал на первой итерации иначе, зная требование сейчас?
- Какие тесты ты добавил, чтобы поймать регрессию при рефакторинге? Что они проверяют — поведение «как было» или «как должно по ТЗ»?
- Покажи замер до и после. Почему именно эта правка дала прирост?
- Если бы ты переписал с нуля — что бы ты выиграл и что потерял?

**Учебная нагрузка по трекам.** На треке C (4 недели) — одна «вторая итерация» в финале программы как контрольная точка (возврат к задаче недели 1 с изменением условия). На треке B (индивидуальный темп) — одна «вторая итерация» через 2–3 месяца после первой сдачи, выборочно по решению наставника. На треке A (2 года) — несколько раз по ключевым модулям диплома, чтобы к защите код был **многоразовый**, не одноразовый.

**12. Локальная правка вместо архитектурного пересмотра — устойчивый студенческий паттерн, усиливаемый ИИ.** Студент, столкнувшись с задачей, требующей архитектурного решения (выбор модели ошибок, владение ресурсами, покрытие интерфейсов, lifecycle сущностей, стиль работы с БД), применяет точечную правку — добавляет случай в существующую структуру, обходит вместо пересмотра. Это **не свойство работы с ИИ** — это **устойчивая исходная склонность**; ИИ-ассистент **усиливает** её, потому что языковая модель оптимизирована на «следующий правдоподобный токен», не на «архитектурный пересмотр» — это структурное свойство модели.

**Эмпирическая база:**

- **Самостоятельная работа без активного ИИ** — дипломный проект студента трека B, ревью 2026-05-15: из 16 пунктов дефектов 8–10 архитектурного класса (SQL через конкатенацию вместо prepared statements; функция чтения тегов маскирует все ошибки одним возвратом `""` без выбора модели `std::optional<T>` / `OFCondition`; тяжёлый DICOM-объект по значению вместо `unique_ptr`; голое `new` в Builder без RAII; CLI не пробрасывает методы Database; нарушение нормативных требований стандарта; отсутствие lifecycle-операций БД). Остальные ~6 — локальные баги (опечатка, забытая конфигурация в одном из режимов сборки, недосмотр рекурсивного обхода).
- **Работа с активным ИИ-ассистентом** — кандидат на стажировку, неделя 2026-05-16..22, 8 проявлений: STL-однострочник перенесён в обёртку модуля; `throw std::invalid_argument` вместо `std::expected` в финальной задаче (хотя `std::expected` взят в предыдущей); C-цикл для I/O вместо STL-алгоритма; дублирование проверки `count==0`; слипшийся try-catch в файле генератора; индексный счётчик через `ssize+cast` вместо `std::ssize`. ИИ предлагал локально — студент применял без архитектурного пересмотра.
- **Слабые проявления** — студентка трека B: пакетный сценарий запуска добавлен рядом с двумя существующими («добавляет, не упорядочивает»); inconsistent seed между уровнями L1/L3/L4/L5 (`random_device` / `chrono` / `12345`); индексный цикл после идиоматизации остался.
- **Исторические наблюдения** — у нескольких предыдущих студентов наставника (без формальных записей, наблюдение из устной памяти многолетней работы). Это самый широкий канал — паттерн устойчиво воспроизводится.

**Причины устойчивости:**
1. **Курсовая логика «решил задачу — сдал — забыл»** не создаёт давления на структуру. На курсовой важно «работает», не «масштабируется» / «обслуживается».
2. **Лектор показывает финальное решение, не процесс архитектурных выборов.** Студент видит «как должно быть», не учится самому формулировать вопрос «какая модель ошибок здесь нужна», «кто владеет этим ресурсом», «как покроется интерфейсом полный жизненный цикл».
3. **ИИ-ассистент по природе предлагает локально.** Языковая модель оптимизирована на правдоподобное продолжение, не на пересмотр структуры. ИИ **усиливает** существующую склонность; не создаёт её.

**Симптомы для диагностики на ревью:**
- **Соотношение архитектурных / локальных дефектов.** Локалист имеет ~60/40 в пользу архитектурного (как у дипломного студента выше); архитект — близко к 90/10 в пользу локальных багов.
- **Реакция на замечание про архитектурный пробел.** Локалист: точечная правка («поменяю на `recursive_directory_iterator`»). Архитект: пересмотр («давай введу `FileReader` с конфигом» / «давай выберу модель ошибок и применю везде»).
- **Какой пункт AI-чек-листа студент игнорирует первым.** Обычно — архитектурный. Это **первый сигнал** склонности.

**Решения:**
1. **«Вторая итерация задачи»** (принцип 11) — главный диагностический инструмент. На возврате через 2–4 недели паттерн виден ясно: локалист патчит, архитект пересматривает структуру под новые требования.
2. **Стартовый промпт сессии AI** (расширение § 4.2 студенческого `AGENTS.md`): «Перед тем как предлагать правку — задай мне вопрос: это локальная правка или нужен архитектурный пересмотр? Если у задачи есть архитектурные оси (модель ошибок, владение ресурсами, покрытие интерфейсов, lifecycle, стиль работы с БД) — назови их **до** правки». Это активирует принцип 12 в сессии без напоминания студенту.
3. **Явный вопрос наставника на ревью**: «Какие архитектурные решения ты в этой задаче принял? Можешь назвать оси?». Типичный студент в первый раз не может назвать ни одной — это диагностика.
4. **Демонстрация при разборе своего же дефекта.** Если на ревью даём список дефектов, отдельно для локальных и архитектурных — студент видит соотношение. Это конкретный сигнал «куда расти».

**Связь с другими принципами:**
- **Принцип 5 (AI)** — *пассажир* и *критический пользователь* при работе с локальным предложением AI ведут себя одинаково (применяют как есть); *соавтор* видит «это локальная правка, не архитектурный пересмотр» — задаёт уточняющий вопрос.
- **Принцип 7 (AGENTS.md как runtime config)** — без процедурной активации в стартовом промпте принцип 12 остаётся декларацией. Студент не «выучивает» склонность к архитектурному мышлению через чтение AGENTS.md.
- **Принцип 11 (Эволюция работающего кода)** — диагностический контекст для принципа 12. На «второй итерации» локалист упирается в потолок, архитект — пересматривает.

**Учебная нагрузка по трекам для принципа 12.** На треке A (2 года) — постепенное наращивание архитектурных навыков через ключевые модули диплома; диагностика по соотношению дефектов на ревью. На треке B (индивидуальный) — выявление склонности по дипломному коду + «вторая итерация» по принципу 11. На треке C (4 недели) — диагностика на КТ-1..КТ-4 (соотношение дефектов) + «вторая итерация» в финале.

**13. Верификация выхода ИИ — по классу артефакта.** Рефлекс независимой проверки того, что выдал ИИ, у студентов не формируется сам — и проявляется одинаково на разных треках, меняется только класс артефакта. Эмпирическая база (2026-06): три студентки гуманитарно-теоретического этапа — сданы галлюцинированные источники (отчёт с будущим годом издания, несуществующий домен вместо реального), не замечена подмена классических первоисточников области русскоязычным обзорным пулом, количественные утверждения проходят сессии без единой просьбы проверить первоисточник; зеркало на C/C++-треке — выдуманные сигнатуры API (давно известный класс, закрыт правилом «сверь по cppreference»).

Правило-таблица «класс артефакта → канал проверки»:

| Класс артефакта от ИИ | Канал верификации |
|---|---|
| Источник / ссылка на статью | Открыть DOI / Google Scholar / eLibrary — статья существует, автор утверждает именно это |
| Число, факт, формула | Первоисточник (не пересказ ИИ) |
| Сигнатура / поведение API | Официальная документация (cppreference, vendor docs) |
| Команда инструмента | `--help` / README (принцип 9) |
| Внешний компонент / версия | Страница релизов проекта |

Фильтр (не каждый чих): проверяется всё, что **уходит в сдаваемый артефакт** — текст работы, код, список литературы; черновое обсуждение проверки не требует. Внимание: декларативная форма этого правила («проверяй факты») в методичках уже была — и не работала; по принципу 7 правило живёт только как процедура: пункт финального прохода перед сдачей (открыть каждый DOI — это действие, не вопрос к себе) и/или строка стартового промпта AI-сессии («на каждый источник и цифру давай ссылку на первоисточник и помечай, что её нужно открыть»).

**14. Предстартовое ревью артефактов кандидата — шаг онбординга.** До выдачи программы запросить у кандидата ссылки на его публичные репозитории / проекты; ~час ревью даёт калибровку, которую анкета и собеседование не дают. Что извлекается: (а) **реальный уровень против самооценки** — самоотчёт ненадёжен в обе стороны: сильный кандидат занижает («владею базовыми навыками» при коде уверенного junior-плюс), осознанный пассажир честен и при этом бесперспективен (принцип 5, self-awareness); калибровка только по артефактам; (б) **мотивационная зона** — домен его собственных проектов подсказывает, на каком материале строить задачник (кейс 2026-06: задачи курса заменены на геометрию точек по CAD-фону кандидата); (в) **базовая линия паттернов до старта** — проявления принципа 12 в его собственном коде (принятый as is сгенерированный монолит, дублирующий 70 % существующей логики) дают чистую точку отсчёта для оценки динамики; (г) **заготовки вариаций контрольных точек по его собственному коду** — повтор/рефакторинг своего же кода без ИИ диагностичнее искусственных упражнений. Сквозные пробелы, видимые из ревью (нет тестов ни в одном репо; сборка только кнопкой IDE), сразу настраивают акценты программы.

**Self-review перед коммитом — обязательное правило.** Перед каждым `git commit` / `git push` студент проходит `git diff` пофайлово и проверяет: не залип ли мусор (артефакты сборки, временные файлы, чужие правки, незавершённые комментарии); внесены ли все исправления по последнему ревью; собирается ли проект; зелёные ли тесты. Это привычка, которую дешевле выработать с первого коммита, чем переучивать потом — у студентов, у которых её не выработали с самого начала, неаккуратность при коммитах остаётся годами.

**Стартовый shell на Windows — PowerShell** (Developer PowerShell for VS как штатный ярлык от Visual Studio). Классический `cmd` — там, где нужна совместимость со старыми сценариями или того требует конкретный инструмент. Подробности и примеры — в строках «Командная строка (CLI)» и «Подготовка окружения» таблицы навыков ниже.

**Для соискателей в Inobitec** параллельно с этим курсом используется доменно-специфический набор задач — `inobitec-practice-tasks/` (отдельный каталог рядом с `swe-curriculum-builder`). Там 5 малых задач в стиле задачника (с лестницами слоёв до предела освоения навыка) и 1 большая задача-проект «Архив медицинских данных», которая через ~20 слоёв инфраструктуры дорастает до production-подобного фрагмента в декорациях продуктов компании (DCMTK, SQLite, spdlog, Crashpad, OpenTelemetry, Lua, CPack, и т. п.). Такой набор сокращает onboarding кандидата при трудоустройстве: к собеседованию у него готовое портфолио мини-проектов, использующих тот же стек, что в реальной кодовой базе компании.

| **Организация заголовочных файлов** | Две связанные дисциплины, обе вырабатываются с первого коммита. **(1) Защита от повторного включения — дифференцированный стиль по языку.** Согласуется с принципом «C на C-стиле, C++ на C++-стиле»: **`.h` (C-заголовки) → классические `#ifndef`-guards** с **уникальным префиксом проекта** (`MYPROJ_STATS_CORE_H`, не просто `STATS_CORE_H` — иначе коллизия с другими либами); C-стандарт `#pragma once` не стандартизирует, C-стиль традиционно строго-портабельный, legacy-кодовые базы и embedded toolchains могут не поддерживать `#pragma once`. **`.hpp` (C++-заголовки) → `#pragma once`**: одна строка вместо трёх, нет коллизий guard-имён, нет риска опечатки в `#define FOO_HPP` / `#endif`, universal-поддержка (GCC ≥3.4, Clang ≥1.0, MSVC ≥2005), фактическая de-facto норма современных C++-стайлгайдов (Google, LLVM, cppbestpractices). Multiple-inclusion-оптимизация компилятора одинакова для обоих вариантов — производительности компиляции выбор не меняет. Классические `#ifndef`-guards на C++-стороне остаются строго-стандарт-портабельной альтернативой (C++ Core Guidelines SF.8 за переносимость) и единственным надёжным вариантом, если в C++-проекте есть копии одного заголовка в разных путях ФС (vendored deps, monorepo). **Главное правило слоя — единый стиль для каждого типа файла на весь репо**: смешение «два разных стиля на `.h`-файлах» или «два разных стиля на `.hpp`-файлах» — сигнал «писал как AI подсказал в этот раз». Допустимый шов — **между** типами файлов (`.h` с guards, `.hpp` с `#pragma once` одновременно в одном репо). **(2) Порядок `#include` директив в `.cpp`** (Google C++ Style Guide, секция «Names and Order of Includes»): группами с пустой строкой между ними — (а) свой соответствующий header первым (если есть, например в `foo.cpp` первым `#include "foo.hpp"` — это **тест на self-containment** этого header, не стилистическая прихоть); (б) C system headers (`<unistd.h>`); (в) C++ standard library (`<vector>`, `<iostream>`); (г) сторонние библиотеки (`<boost/...>`); (д) проектные headers (`"my_module.hpp"`). **Каждый `.cpp` явно включает все используемые им хедеры** — не полагается на транзитивные, иначе рефакторинг чужого хедера ломает файл, который не правили. **Куда движется индустрия — C++20 modules** (`import std;` вместо `#include <iostream>`). Modules отменяют обе техники защиты заголовков и сам механизм текстуального включения. На 2026 поддержка ещё неровная (GCC 14+, MSVC, Clang догоняет) — для большинства проектов остаётся практичный дефолт «`.h` с guards + `.hpp` с `#pragma once`» + правильный порядок include'ов; modules упоминаются как направление, реально подключаются на проектах с зрелой инфраструктурой. | `#pragma once` (директива препроцессора, для `.hpp`); `#ifndef GUARD_NAME` / `#define` / `#endif` (для `.h`); `include-what-you-use` (IWYU) — статический анализатор корректности include'ов, ставится через apt/vcpkg; ссылки — Google Style: <https://google.github.io/styleguide/cppguide.html#Names_and_Order_of_Includes>; Core Guidelines SF.8 / SF.10 / SF.11: <https://isocpp.github.io/CppCoreGuidelines/CppCoreGuidelines#sf-files>; `#pragma once` reference: <https://en.cppreference.com/w/cpp/preprocessor/impl>. C++20 modules: <https://en.cppreference.com/w/cpp/language/modules>. | Все `.h` в репо защищены `#ifndef`-guards с уникальным префиксом проекта; все `.hpp` — `#pragma once`; смешения стилей **внутри одной категории** нет. Каждый `.cpp` имеет порядок include'ов: свой header → blank line → системные группой → blank line → проектные. Студент может объяснить, **почему** разные стили для `.h` и `.hpp` (C-стандарт строго-портабельный + legacy, C++-стиль прагматический + de-facto норма), **почему** свой header идёт первым (self-containment test), **почему** проектные после системных (защита от транзитивных зависимостей), **что такое** IWYU и зачем его в CI. Знает про C++20 modules как направление, понимает, что они отменяют include-механизм. |
| **Первый код и компиляция из консоли** | Сборка «Hello, world!» на C и C++. Раздельно: компиляция/линковка. **Стандарт по умолчанию — современный** (`/std:c++latest` для MSVC, `-std=c++23` для GCC/Clang) — даёт `std::print`, `std::expected`, `<mdspan>`, multi-dim subscript и другие лаконичные конструкции последних стандартов. | MSVC: `cl /nologo /W4 /EHsc /MD /std:c++latest hello.cpp`. GCC/Clang: `g++ -Wall -Wextra -std=c++23 hello.cpp -o hello.exe` (Microsoft Learn, gcc.gnu.org). | Бинарник запускается, понимаешь этапы компиляции/линковки (gcc.gnu.org). |
| **Базовый I/O в C++** | Стандартные потоки `std::cin` / `std::cout` / `std::cerr` как первая работа программы с миром. **`std::getline` vs `std::cin >>`** — типовая ловушка: `cin >>` оставляет `\n` в буфере, следующий `getline` читает пустую строку. Проверка ошибок ввода — `cin.fail()` / `cin.clear()`. **Идиоматический C++23 — `std::print` / `std::println`** для печати с форматом, **`std::format`** (C++20) для подготовки строки; устаревший C-стиль `printf("%s=%d\n", ...)` и каскад `cout << ... << ...` уступают `std::println("{}={}", name, value)`. Причина: однобайтная кодировка совпадает между «байтом в `char[]`» и «символом», `<cctype>` (`std::isupper`, `std::tolower`) и `<ctype.h>` при настроенной локали корректно работают на не-ASCII буквах, посимвольный разбор учебных задач делается без машинерии. Аргументов два, в порядке остроты: (1) **логический** — UTF-8 многобайтная, ломает `<cctype>` на не-ASCII, требует `std::u8string` / wide strings / ICU, посимвольный разбор задач 1-й недели становится дорогим; (2) **Windows-специфичный практический** — на Windows **интерактивный ввод через console handle в режиме `chcp 65001` нестабилен для не-ASCII**: известный многолетний баг моста Windows CRT ↔ console приводит к тому, что `fgets`/`std::getline` из console при cp65001 для русских букв возвращает пустую строку (байты теряются между cmd и stdin программы). На pipe-канале (`program.exe < input.txt`) этого нет — там stdin это файл, CRT читает байты без console-конверсии, тесты могут проходить при сломанном живом запуске. На Linux проблема отсутствует, но KOI8-R на Linux выбираем по логическому аргументу (1) — методическое единообразие между платформами. Эти два потока сообщений нельзя смешивать: если лог-сообщение исчезает после фикса бага — это была отладочная печать (была неправильно оформлена как лог); если временная `cerr`-вставка остаётся после коммита и засоряет `stderr` пользователю — это маскирующийся под лог отладочный код, ремонтируется удалением. | Уровень 0: `std::cerr`, `std::ofstream`. Уровень 1: `spdlog` (header-only / compiled, кросс-платформенно через vcpkg), `Boost.Log`, `log4cpp`. | Уровень 0 присутствует с первой задачи (программа выводит ход работы в `stderr` или файл). Уровень 1: подключена библиотека, в коде используется по назначению (асинхронные синки, ротация если нужна), выбор сделан по чек-листу § 2.1.E. | **Версионирование релизов и совместимость API/ABI** | Дисциплина выпуска: **SemVer** (MAJOR.MINOR.PATCH) с понятным правилом «когда что инкрементируется» — MAJOR при breaking change публичного API, MINOR при добавлении функциональности с обратной совместимостью, PATCH для багфиксов. **Версия зашита в бинарь**, а не живёт только в git-теге: `project(MyProj VERSION 1.2.3 LANGUAGES CXX)` в CMake → автогенерируемый `Version.h` через `configure_file` → флаг `--version` у CLI программы выводит `MyProj 1.2.3+abc1234` (где `abc1234` — git short hash из `git describe --always --dirty`). **CHANGELOG.md** в формате [Keep a Changelog](https://keepachangelog.com/) — каждая версия = свой раздел с подразделами Added / Changed / Fixed / Removed; обязательное условие «не выпускаем релиз без записи в CHANGELOG». **Различие API vs ABI** — для C++-библиотек критично: source compat (старый код пользователя компилируется на новой версии) и binary compat (старый бинарь линкуется с новой `.dll`/`.so` без перекомпиляции). Что **ломает ABI**: добавление поля в публичный класс (меняет sizeof и layout), добавление виртуальной функции (сдвигает vtable), изменение сигнатуры публичной функции (другой mangled name), изменение публичной inline-функции или шаблона. Что **не ломает**: добавление новой не-виртуальной функции, добавление перегрузки, изменение приватных деталей за **pImpl**-идиомой. **Версионирование DLL на Windows** — VS_VERSION_INFO в ресурсной секции (`.rc`-файл с `VERSIONINFO` или CMake `set_target_properties(... VERSION 1.2.3 SOVERSION 1)`); на Linux — SOVERSION (`libfoo.so.1.2.3` + symlink `libfoo.so.1`). | CMake: `project(... VERSION ...)`, `configure_file(Version.h.in Version.h)`, `set_target_properties(mylib PROPERTIES VERSION ${PROJECT_VERSION} SOVERSION ${PROJECT_VERSION_MAJOR})`. Git: `git tag -a v1.2.3 -m "Release 1.2.3"`, `git describe --always --dirty`. Windows: ресурсный файл с блоком `VERSIONINFO`, проверка через `(Get-Item .\app.exe).VersionInfo` в PowerShell. (2) `mytool.exe --version` печатает версию из git, совпадающую с тегом. (3) В корне репозитория есть `CHANGELOG.md` с записью под текущую версию. (4) Студент может на конкретном куске своего кода показать ABI break (например, добавил поле в публичный класс) и обратимое source-совместимое изменение. (5) Для проекта с собственной DLL — версия читается через `Get-Item .\my.dll \| Select-Object -ExpandProperty VersionInfo` и совпадает с `PROJECT_VERSION`. (6) Знает, почему добавление виртуальной функции в библиотеку с миллионом загруженных копий — это всегда `MAJOR`, даже если старый код «выглядит совместимым». |

| **Ускорение сборки: PCH, unity-builds, кеш компиляции** | Когда проект разрастается до десятков `.cpp` с включениями STL/Qt/Windows.h, full rebuild начинает занимать минуты. Прекомпилированные заголовки (PCH) — основной приём: компилятор один раз парсит «тяжёлый» набор заголовков, сохраняет результат во внутреннем формате (`.pch` MSVC, `.gch` GCC, `.pch` Clang) и переиспользует во всех TU. Что класть в PCH: только редко меняющиеся заголовки (стандартная библиотека, сторонние SDK), не свои часто-правимые. **Подводный камень:** TU, собранная через PCH, может «случайно» получить транзитивные включения из PCH — забытый `#include <vector>` в `.cpp` не сломает локальную сборку, но сломается на машине без этого PCH. Лекарство — `include-what-you-use` (IWYU) в CI. Смежные приёмы: **unity-builds** (`CMAKE_UNITY_BUILD=ON` — склейка нескольких `.cpp` в одну TU, ускоряет full rebuild, но ломает анонимные namespace и `static`-функции); **`ccache` / `sccache`** (кеш по хешу препроцессированного исходника + флагов — спасает после `git checkout` между ветками). | CMake: `target_precompile_headers(mytarget PRIVATE <vector> <string> "MyHeavyHeader.h")`. GCC: `g++ -x c++-header pch.h -o pch.h.gch`. `ccache --show-stats` для проверки cache hit rate. | На своём CMake-проекте включил PCH (через `target_precompile_headers`), замерил `cmake --build . --target clean && cmake --build .` до и после — есть документированное ускорение. Может объяснить, почему PCH вреден на мелком проекте (overhead создания PCH > экономии) и почему опасен без IWYU. Для бонуса — включил unity-build и убедился, что тесты проходят (или нашёл конфликт `static`-функций между склеенными TU и его починил). |

| **End-to-end тестирование CLI: acceptance через перенаправление, генераторы, матричный suite** | Юнит-тесты проверяют отдельные функции; CLI-утилиту нужно проверять **целиком — запускать exe с конкретными аргументами и сравнивать stdout/stderr/exit-code с эталоном**. Без генератора, без фреймворка. **(А) Генератор + сравнение для массовых конфигураций.** Подключается, когда руками собрать вход тяжело: десятки тысяч и более чисел/строк, или нужен систематический перебор размеров. Сценарий: `generator.exe \| program.exe > actual.txt && fc actual.txt expected.txt`. **(Б) Матричный acceptance-тест через GoogleTest + boost::process.** Тест-suite на C++ запускает свой же экзешник через `boost::process::child`, ловит stdout/stderr/exit-code, сверяет с эталоном через стандартные `EXPECT_EQ`. Параметризованные тесты (`INSTANTIATE_TEST_SUITE_P`) дают десятки сценариев одной структурой: программа без аргументов → usage в stderr, exit code != 0; `--help` → справка в stdout, exit code 0; неверный аргумент → ошибка в stderr с кодом возврата; разные размеры входа; разные комбинации опций. Уровень (А): `std::mt19937` + `std::uniform_int_distribution` для генератора. Уровень (Б): **GoogleTest + `boost::process`**, запуск через `ctest`. Альтернативы boost::process: `subprocess.h` (header-only, https://github.com/sheredom/subprocess.h), `reproc`. (А): для утилит с массовым входом есть генератор и эталон, тест-сценарий запускается одной командой, воспроизводим (фиксированный seed). Покрыты корнер-кейсы. | **Пакетные менеджеры (vcpkg и системный)** | Два разных инструмента — две роли. **Системный пакетный менеджер ОС** (`apt` на Ubuntu/Debian, `dnf` на Fedora, `brew` на macOS, `choco`/`winget` на Windows) — для toolchain'а и системного окружения: компилятор, отладчик, файловый менеджер, git, базовые утилиты. **`vcpkg` (manifest-mode, `vcpkg.json` рядом с проектом)** — для C++-библиотек проекта (GoogleTest, `boost::process`, OpenJPEG, и т. п.). Понимать сценарий «**мало памяти / много данных / слабый CPU**» — что меняет выбор алгоритма: external sort вместо in-memory; streaming/online-алгоритмы вместо two-pass; sketching (HyperLogLog, Count-Min) вместо точного подсчёта; trade-off память ↔ время. Программные приёмы: cache locality (data-oriented layout, разница `vector<struct>` и `struct of vectors`), минимизация аллокаций (pre-reserve, object pool), reduction системных вызовов. **Чего не делать на этом слое:** микрооптимизации через ассемблер, оптимизация без измерения. Сначала — выбор алгоритма; на ассемблер уходят лишь когда измерения показывают, что алгоритм правильный, кэш греется, а узкое горлышко — конкретная инструкция. **Два класса сценариев применения:** (а) **оптимизация в первой разработке** — пишу под известный профиль нагрузки, выбираю алгоритм заранее; (б) **эволюция работающего кода под изменение условий** — задача уже работала, но объём данных вырос / требования сменились / профиль нагрузки изменился, и нужно вернуться и переработать. Класс (б) — отдельная методическая компетенция, см. § 2.1.0 принцип 11 «Эволюция работающего кода». Программный инструментарий слоя (Big-O, профайлинг, выбор алгоритма) общий для обоих классов; различается методический процесс: (а) — внутри одной разработки, (б) — через возврат после паузы с риском «переписать с нуля» как антипаттерном. | Big-O анализ; `std::chrono` для замеров; профилировщик (см. строку «Профилирование» ниже). | Можешь объяснить, почему задача на 1 ГБ данных при 64 МБ памяти требует другой алгоритм, чем та же задача на 100 КБ. Делаешь измерения до изменения и после, не оптимизируешь на глаз. Не лезешь в ассемблер до того, как алгоритм оптимален. **Также — на «второй итерации задачи»** (возврат к ранее сданной задаче через 2–4 недели при изменении условий) видишь свой код как чужой; минимальной правкой адаптируешь под новое требование, не переписывая с нуля; добавляешь characterization-тесты, если их не было на первой итерации; сверяешь метрики до и после. |
| **Параллелизм и многоядерные вычисления** | В современном C++ многопоточность — стандартная часть языка, не «продвинутая тема». | **Профилирование и производительность** | Бюджет производительности; профилировщики; «горячие точки»; отчёт. **PGO (Profile-Guided Optimization) — программный приём ускорения** на финальной сборке: компилятор оптимизирует горячие пути на основе реального профиля работы. Production-бинарь stripped (без debug-info внутри), `.debug`/pdb отдельно архивирован по версии+SHA1. Знает разницу между статической и динамической линковкой по эффекту на размер, переносимость, обновляемость. Для Linux — понимает glibc-совместимость, проверял запуск бинаря на VM с другой версией Ubuntu. |

#### 2.1.B. Шпаргалка: C++ vs C идиомы

Частая ловушка для студентов с опытом C — писать на C++ так же, как на C. Это рабочий код, но на ревью его придётся переписывать: индустрия ждёт идиоматический C++. Базовая карта соответствий:

| Задача | C-стиль | C++-стиль |
|---|---|---|
| Печать в stdout | `printf("%d\n", x);` | `std::cout << x << '\n';` или `std::print("{}\n", x);` (C++23) |
| Цикл по контейнеру | `for (int i = 0; i < n; ++i) work(a[i]);` | `for (auto& v : a) work(v);` или `std::ranges::for_each(a, work);` |
| Поиск / преобразование / копирование | ручной цикл с индексом | `std::ranges::find` / `transform` / `copy` |
| Динамический массив | `T* p = malloc(n * sizeof(T)); … free(p);` | `std::vector<T> v(n);` |
| Строки | `char buf[256]; strcpy / snprintf(...)` | `std::string s; std::format("{}…", …);` |
| Обработка ошибок | коды возврата + `errno` | исключения / `std::expected` (C++23) / `std::optional` |
| Передача массива в функцию | `func(T* p, size_t n)` | `func(std::span<T> v)` |
| Парсинг аргументов | `argv[i]` руками | `boost::program_options` / `CLI11` |
| Защита от двойного включения | `#ifndef HEADER_H ... #define ... #endif` | `#pragma once` (поддерживается всеми основными компиляторами) |
| Перечисления | `enum { RED, GREEN };` | `enum class Color { Red, Green };` |
| Полиморфные данные | `void*` + теги-дискриминаторы | шаблоны / `std::variant` / `std::any` |
| Управление ресурсом (файл, мьютекс, память) | ручные `init` / `destroy` | RAII (деструктор) + умные указатели (`unique_ptr` / `shared_ptr`) |
| «Безопасный» каст | `(T)x` C-style cast | `static_cast<T>(x)` / `dynamic_cast<T>(x)` / `reinterpret_cast<T>(x)` |
| Константы | `#define MAX_N 100` | `constexpr int kMaxN = 100;` |
| Многомерный индекс | `arr[i][j]` (требует 2D-array или указатель на указатель) | `arr[i, j]` для классов с `operator[](size_t, size_t)` (C++23); для view над линейным массивом — `std::mdspan` (C++23) |
| Обход ассоциативного контейнера | `for (auto it = m.begin(); it != m.end(); ++it) use(it->first, it->second);` | `for (const auto& [k, v] : m) use(k, v);` (structured bindings, C++17 — но всё ещё многие пишут по-старому) |
| Преобразование диапазона в контейнер | ручной цикл / `std::vector<T>(rng.begin(), rng.end())` | `auto v = rng \| std::ranges::to<std::vector>();` (C++23) |
| Бесконечность с плавающей точкой | `INFINITY` (`<math.h>`, макрос) | `std::numeric_limits<double>::infinity()` (`<limits>`) — типизирована, работает с любым FP-типом |
| Ширина числа в символах (для выравнивания таблицы) | `int w = 1; while (n >= 10) { n /= 10; ++w; }` — ручной подсчёт | `auto w = std::format("{}", n).size();` или `std::formatted_size("{}", n)` (без аллокации) |
| Возврат строкового литерала из функции | `std::string get_label() { return "лет"; }` — конструируется временная `std::string` на каждый вызов | `std::string_view get_label() { return "лет"; }` — указатель+длина, без аллокации |
| Минимум, максимум, сумма за один проход | три цикла подряд / один цикл с тремя редукциями вручную | `auto [min_it, max_it] = std::ranges::minmax_element(v);` + `auto sum = std::ranges::fold_left(v, 0.0, std::plus{});` — два прохода, но идиоматичных; для одного прохода — кастомный `fold` |
| Размер контейнера как `int` (например, для индекса) | `(int)v.size()` | `static_cast<int>(v.size())` (явный narrowing); ещё лучше — `std::ssize(v)` (C++20), возвращает signed-size без cast |
| Парсинг числа из argv | `int k = atoi(argv[1]);` — без проверки ошибки | `auto k = std::stoi(argv[1]);` (бросает исключение); или `int k; std::from_chars(argv[1], argv[1]+strlen(argv[1]), k);` — без исключений (C++17) |

Идиоматический C++ — это **STL-алгоритмы + RAII + строгая типизация**, а не «C с классами». На ревью наставник смотрит, ушёл ли студент от C-привычек, а не только корректность результата: код в C-стиле, который «работает», всё равно требует переписывания, потому что его не сможет поддерживать команда, привыкшая к C++.

**Типичный анти-паттерн «C++ как C с `std::print`»** — взять C-код, заменить `printf` на `std::print` и оставить всё остальное (индексные циклы, ручные буферы, C-cast, `INFINITY`, `(int)size()`). Это **рабочий код**, но он не C++: тело функций остаётся в C-логике, STL подключена только в одной точке (вывод/ввод). На ревью такой код **переписывается целиком** — это не косметика, это «не тот язык».

**Рутинный приём «проверка идиоматичности после написания»** — после того как версия задачи работает, перед коммитом — отдельный запрос к ИИ-помощнику: «проверь, нет ли в этой функции C-наследия (C-style cast, ручные циклы вместо STL-алгоритмов, C-макросы вместо `constexpr`/`std::numeric_limits`, ручные буферы вместо `std::format`); покажи конкретные строки и предложи замену». Работает потому, что современные модели знают идиоматичный C++ лучше, чем студент-новичок; задача — научиться **искать собственное C-наследие в коде**, ИИ — инструмент. Это критический подход (см. § 1.2 в [01-teoriya.md](01-teoriya.md) — анти-подхалимажные приёмы): не «перепиши за меня», а «помоги увидеть, что я бы не увидел сам».

**Детальная шпаргалка по обработке ошибок** (расширение строки «Обработка ошибок» выше). Полная архитектура обработки ошибок — в § 2.1.F «Архитектура обработки ошибок»; здесь — карта быстрых соответствий «C → C++» для типовых ситуаций.

| Ситуация | C-стиль | C++-стиль (основной — `std::expected` / `std::optional`) |
|---|---|---|
| Функция может вернуть ошибку прикладной логики | `int parse(const char* s, int* out);` — return code: 0=успех, !=0=ошибка | `std::expected<int, ParseError> parse(std::string_view s);` (C++23) — тип возврата явно говорит «может вернуть ошибку» |
| Функция может не найти значение (поиск/lookup) | `T* find(Container*, Key);` — `NULL`=не найдено | `std::optional<T> find(const Container&, Key);` — без аллокации |
| Парсинг числа из строки | `int x = atoi(s);` — без проверки; `errno` + `strtol` — с проверкой | `std::from_chars(s.data(), s.data()+s.size(), x);` — error_code в out-параметре, без исключений |
| Композиция операций с ошибками | проверка result после каждого вызова: `if (rc1) return rc1; rc2 = ...; if (rc2) return rc2;` | `expected.and_then(step1).and_then(step2).or_else(handle)` — monadic-style (C++23) |
| Контракт функции «должно быть выполнено предусловие» | `assert(p != NULL);` — выключается в Release | Debug: `assert(p)`. Release: `std::expected<T, E>` если ошибка штатная; явный `throw std::logic_error` если нарушение инварианта; C++26 Contracts (`pre(p)`, `post(...)`) — направление развития |
| Exit code из main | `return 1;` — недифференцированно | `return EXIT_FAILURE` или `return 65;` (sysexits: `EX_DATAERR`); двухканальный путь: exit code = категория для shell, stderr = детализация через `errno`/`ec.message()` |
| Класс ошибки ввода (что именно сломалось) | `if (scanf(...) != 1) ...` — не отличает EOF от не-числа от IO-ошибки | `cin.eof()` / `cin.fail()` / `cin.bad()` после операции; для типизации — `std::error_code` с категорией |
| IO-ошибка в декларативном pipeline (`ranges::copy(view, sink)`) | проверка после каждой операции через `ferror(stream)` | `cin.exceptions(std::ios::badbit)` — только `badbit`, не `failbit` (EOF штатный); `try { ranges::copy(...); } catch (const std::ios_base::failure& e) { auto ec = e.code(); ... }` |
| Сообщение об ошибке пользователю | `fprintf(stderr, "Error: %s\n", err);` — обобщённый штамп | `std::println(stderr, "не удалось обработать поток: {} (errno {})", ec.message(), ec.value());` — конкретика (см. § 2.1.C) |

**Антипаттерн в C++-стеке курса — исключения как основная модель control flow.** В курсе для C++ выбрана двухмодельная схема: основная — `std::expected` / `std::optional` / коды возврата; ознакомительная — исключения (механизм, exception safety, RAII, `noexcept`, обёртки для C-ABI). Обоснование и лестница моделей по неделям — § 2.1.F. Применять исключения как **основную модель** в коде курса — методический анти-паттерн.

#### 2.1.C. Стиль диагностических сообщений

Сообщения программы пользователю — не «Ошибка» и не «Упс», а информация: что не получилось, какое значение получено, какое допустимо. Лаконично, без эмоций, без обвинений. Это не косметика: небрежные сообщения ставят программу в категорию «школьная лабораторка» с первого взгляда, ещё до анализа кода. Современные программисты массово игнорируют эти правила — отсюда заметная часть негативного впечатления от чужого кода, которое формируется ещё до разбора алгоритмов.

1. **Конкретика, не констатация.** Плохо: `Ошибка, число не в диапазоне`. Хорошо: `N должно быть в [1; 20]; получено: 42`.
2. **Без префикса «Ошибка:» / «Error:».** То, что это сообщение об ошибке, ясно из stderr и exit code != 0. Префикс — шум.
3. **Без обвинения пользователя.** Не «вы ввели неправильно», а «значение должно быть числом в диапазоне».
4. **Без эмоций.** Не «Упс!», не «К сожалению…», не «Извините». Программа сообщает факт, а не извиняется.
5. **GNU-формат для CLI:** `program: <текст>`. Стандарт компиляторных сообщений (gcc, clang, MSVC выдают `file:line:col: error: text`); упрощает grep по логам и парсинг в скриптах.
6. **stderr + exit code != 0.** stdout остаётся чистым для рабочего вывода — это критично для перенаправлений `> output.txt` и для пайплайнов `program | другой_program`. Exit code — это **категория** ошибки (см. sysexits в § 2.1.F и в строке «UX и справка командной строки» § 2.1.A); stderr — **детализация** для человека/лога. Каналы несут разную информацию, не миксовать.
7. **Детализация через `errno` и `ec.message()`.** Когда ошибка приходит от системного вызова или потокового исключения, не выводить `e.what()` в одиночку — это даёт штамп вида `"basic_ios::clear: iostream error"`, не информацию. Правильно — типизированный `std::error_code` с человекочитаемым `.message()` и численным `.value()` + контекст операции: `std::println(stderr, "не удалось обработать поток stdin: {} (errno {})", ec.message(), ec.value());`. `ec.message()` даст `"Broken pipe"` / `"No space left on device"` / `"Permission denied"` — конкретику, по которой пользователь понимает, что чинить.

**Антипаттерны, ловящиеся одной grep-проверкой по diff'у** (для pre-commit / CI):
- `grep -nE 'Ошибка|Error[^=:]|Failed|Не удалось' diff.txt` — все шаблонные формулировки на пересмотр (некоторые легитимны, например `Error:` в GNU-формате компилятора, но большинство — штампы без конкретики).
- `grep -nE 'e\.what\(\)' diff.txt | grep -v 'ec\.message'` — `e.what()` в одиночку без обрамляющего контекста операции — почти всегда штамп.

Это **детерминистическая проверка** — дешевле и надёжнее, чем «не забыть применить правило на ревью». См. § 2.1.0 принцип 7 «AGENTS.md как runtime config» — каждое правило сопровождается процедурой.

Локализация — по системной локали ОС (см. § 2.1.A). Английский — fallback, если локаль не русская.

**На ревью** наставник смотрит сообщения об ошибках первым делом — это самый дешёвый сигнал, что код писался с уважением к пользователю или для галочки. **Связь с § 2.1.F** «Архитектура обработки ошибок»: стиль сообщений из этого раздела — это **детализация в stderr-канале**; полная архитектура (exit codes, классификация ошибок ввода, `std::expected`, исключения как механизм, постмортем) развёрнута там.

- **Microsoft Writing Style Guide → «sorry»** (когда уместны извинения, когда нет) — <https://learn.microsoft.com/en-us/style-guide/a-z-word-list-term-collections/s/sorry>

#### 2.1.D. Режимы сборки

Сборка — не «выбери один из четырёх режимов и сиди в нём», а **последовательность подключения** по мере роста задачи. Принцип согласуется с § 2.1.0 пунктом 6 (голая версия с первого дня, промышленная позже): новый режим подключается, когда **у задачи появилась мотивация**, не «сразу всё на всякий случай». Четыре ступени:

##### Ступень 1. Debug + ASan + UBSan — стартовый режим (с 1-й задачи)

**Сообщаем студенту явно:** «работаем в Debug, потому что в нём активные runtime-проверки находят баги в момент возникновения, а не задним числом после тестов или у пользователя». Это **единственный режим на 1-й неделе** — никаких «двух сборок параллельно».

Флаги:
- **GCC/Clang:** `-O0 -g -fno-omit-frame-pointer -fsanitize=address,undefined`
- **MSVC:** `/Od /Zi /fsanitize=address` (UBSan нет в MSVC, частично заменяется `/analyze` и `/RTC`)

Что включено и почему:
- `-O0` / `/Od` — без оптимизации; исходник 1:1 с исполнением, отладчик идёт по строкам, переменные читаются предсказуемо.
- `-g` / `/Zi` — debug-info для отладчика.
- **ASan** — out-of-bounds, use-after-free, утечки (через интегрированный LSan); ~2× замедление.
- **UBSan** — signed overflow, null-deref, нарушения выравнивания, /0, неверные касты; ~10–20% замедление; на MSVC недоступен.
- iterator debugging в STL: libstdc++ `_GLIBCXX_ASSERTIONS` / libc++ `_LIBCPP_DEBUG` / MSVC `/MDd` или `/MTd` — STL-контейнеры проверяют границы и валидность итераторов.

**Бенчмарки на Debug-сборке не делать** — Debug медленнее в ~5–20× за счёт `-O0`, санитайзеров и проверок STL. Для замеров есть Release (ступень 3).

**Варианты Debug под конкретные сценарии** (опциональные ответвления):
- **TSan** (`-fsanitize=thread`, GCC/Clang) — когда в задаче появилась многопоточность; ~5–15× замедление; **не сочетается с ASan**.
- **MSan** (`-fsanitize=memory`, только Clang) — чтение неинициализированной памяти; ~3× замедление; требует пересборки всех зависимостей под MSan.

##### Ступень 2. Coverage — добавляется, когда появились тесты

Когда введён слой тестирования (юнит-тест как exe или GoogleTest) — **отдельная сборка для измерения покрытия**. Это не «переключатель внутри Debug», а отдельный артефакт со своей сборкой; coverage и Debug-санитайзеры не сочетаются — держим разными job'ами.

Двухпроходная схема: компилятор пишет `.gcno` (структура программы), runtime пишет `.gcda` (счётчики basic blocks).

| Шаг | GCC/Clang | MSVC |
|---|---|---|
| Сборка с инструментированием | `--coverage` (= `-fprofile-arcs -ftest-coverage`); типично `-O0` или `-O1` | `Microsoft.CodeCoverage.Console` (динамическое инструментирование, оптимально `/O2 /Ob1`) |
| Прогон тестов | Запуск как обычно → `.gcda` рядом с `.gcno` | `Microsoft.CodeCoverage.Console collect -- <test>.exe` |
| Текстовый отчёт | `gcov source.cpp` | встроенный в VS |
| HTML-отчёт | `lcov --capture --directory . --output-file cov.info && genhtml cov.info --output-directory cov-html` | Visual Studio либо экспорт Cobertura |
| Альтернатива на Clang | `-fprofile-instr-generate -fcoverage-mapping` → `llvm-profdata merge` → `llvm-cov show/report` | OpenCppCoverage |

Стандартный формат отчётов — **Cobertura XML** (совместим с GitHub Actions, GitLab CI, SonarQube, Azure DevOps).

**Не запускать coverage с оптимизациями выше `-O1`** — инлайнинг ломает соответствие строк бинарю, отчёт получается «дырявым». Coverage не для production, только для отчёта о покрытии.

##### Ступень 3. Release — добавляется, когда нужно мерить производительность

Появляется на слое «Оценка алгоритмической сложности и адаптация под ресурсы» — бенчмарки через `std::chrono` требуют оптимизированной сборки. **Это ещё не production-артефакт**, это сборка **для замеров**.

Флаги:
- **GCC/Clang:** `-O2 -g -DNDEBUG -Wall -Wextra -Wpedantic`
- **MSVC:** `/O2 /Zi /DNDEBUG /W4` + линкер `/DEBUG`

Что меняется относительно Debug:
- `-O2` / `/O2` — стандартный уровень оптимизации (инлайнинг, разворачивание циклов, удаление мёртвого кода).
- `-DNDEBUG` / `/DNDEBUG` — выключает `assert` и `_GLIBCXX_ASSERTIONS` в STL.
- Стандартная библиотека — Release-CRT (`/MD` / `/MT` на MSVC), iterator debugging выключен.
- **`-g` / `/Zi` остаются** — debug-info генерируется (про split — на ступени 4).
- Sanitizers **выключаются** — несовместимы с замерами производительности и зачастую с `-O2`.

**Что не делать на Release-сборке:** интерактивную отладку (оптимизатор переставляет инструкции, отладчик прыгает между строками); тесты под санитайзерами (несовместимо).

**CMake-режимы.** `Debug`, `Release`, `RelWithDebInfo` (на практике совпадает с современной Release-практикой: `-O2 -g`), `MinSizeRel` (`-Os`). Выбирается через `-DCMAKE_BUILD_TYPE=...` (single-config — Ninja, Makefiles) или `--config ...` (multi-config — Visual Studio).

##### Ступень 4. Production — финальный артефакт для пользователя

Когда проект готовится к раздаче пользователю (последние недели курса) — Release-сборка обрастает обвязкой, превращаясь в production-артефакт. Это **цепочка**, не один флаг:

**4a. Split debug info.** Release собирается с `-g` / `/Zi`, после сборки debug-info отделяется. Пользователю поставляется **stripped-бинарь** (на порядок меньше), отделённый `.debug`/`.pdb` хранится у разработчика и подгружается при разборе crash-дампов от пользователей.

- **MSVC:** автоматически — `/Zi` + линкер `/DEBUG` сразу создают `.pdb` рядом с exe; в самом exe debug-info нет.
- **GCC/Clang:** два шага вручную — `objcopy --only-keep-debug program program.debug && strip --strip-debug program`. Связь stripped-бинаря и `.debug` — через **build-id** (`readelf -n` извлекает).

Pdb / `.debug` **не отдаются пользователю** — раскрытие внутренностей; но **обязательно сохраняются** у разработчика (symbol server, архив по build-id или версии). Без сохранённого debug-файла crash в production = голые адреса в стеке, разобрать нельзя.

**4b. Стратегия поставки: статика vs динамика.** Решение о готовом артефакте для пользователя — **отдельное от техники сборки** (раздельная компиляция и собственные `.a`/`.so` модулей проекта — это § 2.1.A «Этапы сборки», подключается раньше). На production-ступени решается **как программа поставляется**.

| Стратегия | Что значит | Размер | Запуск | Когда выбирать |
|---|---|---|---|---|
| **Полная статика** (`-static` GCC/Clang) | Линкуется всё, включая libc, в один бинарь | Большой (на 5–20 МБ больше) | На любой Linux-машине с совместимой архитектурой **без зависимостей** | Распространение единственным файлом; целевые системы разные / неизвестные; контейнер без libc |
| **Динамика по умолчанию** | Системные библиотеки (libc, libstdc++) подгружаются с целевой машины | Компактный | Требует совместимой версии glibc на целевой машине | Стандартный путь для большинства Linux-дистрибутивов; библиотеки обновляемы через ОС |
| **Частичная статика** (`-static-libgcc -static-libstdc++`) | Статически линкуется только C++ runtime, остальное динамика | Средний | Требует только совместимой libc | Хочется не зависеть от конкретной версии libstdc++, но не раздувать бинарь |
| **MSVC: `/MT` vs `/MD`** | `/MT` статически линкует C runtime; `/MD` динамически (требует VC Redistributable на целевой машине) | Аналогично | Аналогично | `/MT` — single-file дистрибуция; `/MD` — стандартный путь, нужен установщик VC Redist |

**glibc-совместимость (Linux).** Динамический бинарь, собранный против glibc 2.39, **не запустится** на Ubuntu 20.04 с glibc 2.31 (отсутствуют новые символы). Варианты решения: (а) собирать на старой системе с минимальной поддерживаемой glibc; (б) собирать `-static`; (в) использовать [zig cc](https://ziglang.org/) или [linuxdeploy](https://github.com/linuxdeploy/linuxdeploy) для targeting конкретной glibc. Проверка совместимости: запуск собранного бинаря на VM с разными версиями Ubuntu.

**Trade-off, который студент должен понимать:**
- Статика — переносимый exe без зависимостей, **но** большой размер и невозможность централизованного обновления уязвимостей в libc.
- Динамика — компактный exe, обновляемый через ОС, **но** требует совместимой версии библиотек на целевой машине.

Решение по стратегии — **часть проектного дизайна**, не настройка сборки «на глаз».

**Опционально: отложенная загрузка библиотек (для крупного софта).** Когда программа линкована с десятками DLL/`.so` (например, медицинский комплекс с DCMTK / OpenCV / OpenGL), старт может занимать секунды — ОС загружает **все** импорты до запуска `main()`. Если опциональные подсистемы используются не каждой командой, **отложенная загрузка** ускоряет старт: библиотека грузится при первом вызове функции из неё.

- **MSVC:** `/DELAYLOAD:foo.dll` в линкере — генерирует stub'ы вместо немедленной загрузки.
- **GCC/Clang/Linux:** автоматики нет — ручной `dlopen("libfoo.so", RTLD_LAZY)` + `dlsym` для нужных функций; нужно явно писать код-обёртку.

Подключать **только когда измерения показывают долгий старт** (не «на всякий случай»). Для учебных программ курса не нужно — мелкие CLI-утилиты не имеют этой проблемы.

**Историческая справка: base address DLL.** Раньше для ускорения загрузки крупного Windows-софта каждой собственной DLL назначался уникальный preferred base address (`/BASE:0x10000000` в линкере MSVC, опционально из CMake-сценария для production-конфигурации) — если адреса не пересекались, Windows не делала дорогой rebase при загрузке. На **современных x64-системах с ASLR** (Address Space Layout Randomization, стандарт с Windows Vista+) эффект почти нулевой: ОС всё равно рандомизирует базовые адреса при каждом запуске, заданный `/BASE` перезаписывается. Приём остаётся актуальным только для 32-битных целей (legacy embedded, WinXP-совместимость) и для DLL, загружаемых до инициализации ASLR. **Не применять в новых проектах**, но узнать на ревью legacy-кода — встречается. На Linux аналога нет: динамический линкер делает relocation автоматически.

**4c. PGO (Profile-Guided Optimization) — опционально, для CPU-bound.** Двухпроходная сборка: компилятор оптимизирует горячие пути по реальному профилю.

| Шаг | GCC/Clang | MSVC |
|---|---|---|
| (1) Инструментированная сборка | `-fprofile-generate -O2` | `cl /GL /LTCG:PGI` |
| (2) Нагрузочный запуск на репрезентативном входе | `./program <реальная нагрузка>` → пишет `.gcda` | запуск → пишет `.pgd` |
| (3) Финальная оптимизированная сборка | `-fprofile-use -O2` | `cl /GL /LTCG:PGO` |

Типичное ускорение — **5–15% «бесплатно»**, без переписывания кода. Окупается, когда: (а) программа CPU-bound (парсеры, сортировки, числодробилки); (б) есть репрезентативный нагрузочный набор данных; (в) релизный цикл позволяет двухпроходную сборку. На IO-bound коде или при непредставительном профиле прироста нет, в худшем случае пессимизация.

##### Что НЕ комбинируется

- **ASan + TSan** — несовместимы, конфликтуют за инструментирование памяти.
- **Sanitizers + PGO** — `-fprofile-generate` несовместим с `-fsanitize=*`.
- **Sanitizers + coverage** — формально работает, но интерпретация смешанная; держать отдельными job'ами.
- **Debug-CRT + Release-CRT** в одной программе (MSVC) — разные ABI стандартных контейнеров (iterator debugging меняет sizeof `std::vector`); смешивать `.obj`/`.o`-файлы Debug и Release — UB.

##### Матричный CI на финальном проекте

Когда проект готов к раздаче, CI обычно гонит несколько режимов параллельно как разные job'ы — **финальная конфигурация**, не начальная:

| Job | Режим | Когда гонять |
|---|---|---|
| `dev` | Debug + ASan + UBSan | каждый push |
| `release` | Release с `-g` / `/Zi` | каждый push |
| `coverage` | Coverage | каждый push в main или nightly |
| `tsan` | Debug + TSan | для multithreaded-кода; nightly |
| `pgo` | PGO двухпроходная | релизный pipeline, не каждый push |

Путь к этой матрице — через ступени 1–4 по мере роста задачи. Не «вот вам матрица сразу на 1-й неделе», а «вот что добавляется, когда конкретно понадобилось».

#### 2.1.E. Критерии выбора промышленной библиотеки

При переходе с голой версии на промышленную (см. § 2.1.0 принцип 6) выбор между несколькими библиотеками — **отдельная инженерная задача**. Без критериев студент берёт первое предложение AI-помощника или то, что в учебнике. Чек-лист ниже — минимум, который проходится **до** добавления библиотеки в `vcpkg.json` / `pyproject.toml` / `package.json`. На каждый пункт уходит 1–5 минут — на GitHub видно сразу.

**1. Проект жив.** Дата последнего коммита (не «годы заброшено»), частота релизов (регулярные тэги, не один в три года), количество открытых vs закрытых issues. Заброшенный проект = баги останутся вашими навсегда.

**2. Bus factor мейнтейнеров.** Сколько активных мейнтейнеров в команде. **Один человек — риск**: если он уйдёт или выгорит, проект остановится. Два-три и больше — устойчивость. Видно в графе contributors на GitHub за последний год.

**3. Скорость отклика на issues и PR.** Медианное время от issue до первого ответа мейнтейнера (не до закрытия, до **реакции**). Хороший проект отвечает в течение дней-недели; плохой — issues висят месяцами без ответа. Это лучший индикатор «можно ли рассчитывать на помощь, если что-то сломается».

**4. Адопция в индустрии.** Кто использует — крупные известные проекты в каких именах. Для C++: используется ли в Chromium, LLVM, Google, Boost-экосистеме. **Звёзды на GitHub — слабый сигнал** (легко надуть); реальные зависимые проекты — сильный.

**5. Лицензия совместима.** MIT / BSD / Apache 2.0 — почти всегда ок для коммерческого использования. LGPL — ок для динамической линковки. **GPL — заразный**, часто блокер для коммерческих проектов. Проверять **до** интеграции, не после — переписывание стоит дороже первичного выбора.

**6. Документация покрывает non-happy-path.** Не только «как использовать в идеальном случае», но и: что делать при ошибках, какие edge cases описаны, есть ли changelog с описанием изменений по версиям, описаны ли limitations явно. Документация «читай исходник» = overhead растёт линейно.

**7. У самой библиотеки есть тесты и CI.** Открыть репо: есть ли папка `test/`/`tests/`, есть ли CI-конфиг (`.github/workflows/`, `.gitlab-ci.yml`), зелёный ли последний билд. Если у библиотеки своих тестов нет — это сигнал, что библиотеку могут сломать без вашего ведома.

**8. Композабельность API.** Библиотека разбита на логические куски, которые комбинируются под задачу, или это монолит «всё или ничего»? Композабельная — `spdlog` (отдельные синки, форматтеры, уровни); монолитная — фреймворки типа Qt (полностью или никак). Для одной утилиты композабельная почти всегда выигрывает.

**9. Доступность через пакетный менеджер.** Есть ли в `vcpkg` / `Conan` / `apt` / стандартном менеджере вашего стека? Если приходится собирать вручную из исходников и патчить CMake — это **многократно** увеличивает стоимость подключения и поддержки.

**10. Транзитивные зависимости.** Сколько других библиотек тянет? Header-only `spdlog` vs `Boost.Log` (тянет половину Boost) — для одной задачи разные стоимости. Меньше зависимостей = меньше сюрпризов при обновлении.

**11. Совместимость со стандартом и компилятором.** Требует ли C++23/26? Поддерживается ли вашим компилятором (gcc/clang/MSVC + конкретная версия)? Поддерживается ли вашей целевой платформой (Linux glibc / Windows MSVC / Android NDK)? Эти ограничения нужно знать **до** включения в проект.

**Применять в порядке списка.** Если первые 3–4 пункта (жив ли, bus factor, отклик, адопция) дают красные флаги — дальше можно не смотреть, эту библиотеку не берём. Если все 11 зелёные — это **базовая инженерная гигиена**, не гарантия от проблем, но защита от очевидных провалов.

**Что НЕ годится как единственный критерий выбора.** Звёзды на GitHub без поправки на возраст и контекст; «AI-помощник посоветовал»; «в учебнике использовали»; «эту библиотеку я знаю». Каждый из этих сигналов отдельно — слабый.

#### 2.1.F. Архитектура обработки ошибок

Развёртывание принципа 10 («Обработка ошибок — сквозная дисциплина») в § 2.1.0. Здесь — конкретная лестница моделей по неделям курса, антипаттерны, и связь с остальными разделами § 2.1.

**Контекст архитектурного выбора.** Обработка ошибок в C/C++ — один из самых дискуссионных слоёв инженерной практики, потому что язык предлагает несколько моделей одновременно, и индустриальное согласие отсутствует. На одном полюсе — крупные проекты с `-fno-exceptions` (Google C++, LLVM/Clang, WebKit, Chromium, Unreal Engine, embedded, HFT). На другом — Qt-приложения, Boost-based код, бизнес-серверы, где исключения нормальная модель. C++23 ввёл `std::expected<T, E>` как первоклассный механизм типизированных кодов возврата — это **движение стандарта в сторону кодов**, признание что чисто исключения не покрывают все сценарии. Курс делает архитектурный выбор под целевой стек (системный / multi-core / embedded / инструментальный) — основная модель `std::expected`, исключения ознакомительно.

##### Неделя 1 — базовая корректность

**1. Exit codes — дифференцированные, ориентир `sysexits.h` (BSD):**

| Код | Имя | Когда |
|---|---|---|
| 0 | `EX_OK` | Успех |
| 64 | `EX_USAGE` | Плохой аргумент CLI (отсутствует, не парсится, недопустимое значение) |
| 65 | `EX_DATAERR` | Плохой формат входных данных (не-число в потоке, нарушение схемы) |
| 66 | `EX_NOINPUT` | EOF на ожидаемом входе / отсутствие входного файла |
| 71 | `EX_OSERR` | Ошибка ОС (`errno`-категория) |
| 74 | `EX_IOERR` | IO-ошибка (диск, сеть, broken pipe) |

**Антипаттерн — `return 1;` везде.** Внешний пользователь утилиты в pipeline не различает, **что починить**: плохой argv, плохой ввод, нет данных, или сломалось железо.

**2. Классификация ошибок ввода — что именно сломалось:**

| Состояние потока (C++) | Что означает | Код возврата |
|---|---|---|
| `cin.eof()` | штатный конец потока | 0 (если данные были) / 66 (если ожидали данные) |
| `cin.fail()` | не удалось распарсить (например, не-число) | 65 |
| `cin.bad()` | реальная IO-ошибка (диск, broken pipe) | 74 |

На C-стороне аналог через `feof(stream)` / `ferror(stream)` + `errno`. **Антипаттерн — `if (scanf("%d", &n) != 1) return 1;`** — не отличает EOF от не-числа от IO-ошибки, теряется три класса диагностики.

**3. Двухканальный путь exit + stderr.** Exit code — **семантический канал shell** (категория для скрипта-обёртки `case $? in 64) ...; 65) ...; esac`); stderr — **детализация для человека/лога** (`errno` + `ec.message()` + контекст операции). Каналы несут разную информацию, не дублируют друг друга:

```cpp
} catch (const std::ios_base::failure& e) {
    auto ec = e.code();
    std::println(stderr, "не удалось обработать поток: {} (errno {})",
                 ec.message(), ec.value());
    return 74;  // EX_IOERR категория для shell; errno в stderr для человека
}
```

**Антипаттерн — `return ec.value();`** (errno как exit code):
- POSIX-shell exit ограничен 0–255 (`$?` mod 256), с зарезервированными значениями: `126` (не исполняема), `127` (не найдена), `128+N` (убит сигналом N). Errno-значения с этим конфликтуют — `EHWPOISON=133` будет прочитан bash как «убит SIGCONT+5».
- Errno-значения формально не стандартизованы между ОС (POSIX задаёт **имена**, не числа). Скрипт обёртки `if [[ $? == 32 ]]` сломается между Linux/macOS/BSD-emulation.
- Sysexits-категории несут **класс** ошибки (usage / data / io), errno — **механику** конкретного syscall. Это разные семантические уровни; смешивать их через exit code загрязняет канал.

##### Неделя 2 — типизированные модели и исключения как механизм

**1. `std::expected<T, E>` как основа.** Тип возврата явно говорит «может вернуть ошибку», вызывающий обязан обработать. Базовое использование:

```cpp
auto parse_port(std::string_view s) -> std::expected<int, std::string> {
    int value;
    auto [ptr, ec] = std::from_chars(s.data(), s.data() + s.size(), value);
    if (ec != std::errc{} || ptr != s.data() + s.size()) {
        return std::unexpected(std::format("ожидается целое число; получено: \"{}\"", s));
    }
    if (value < 1 || value > 65535) {
        return std::unexpected(std::format("порт должен быть в [1; 65535]; получено: {}", value));
    }
    return value;
}

// callsite:
auto port = parse_port(argv[1]);
if (!port) {
    std::println(stderr, "--port: {}", port.error());
    return 64;  // EX_USAGE
}
// дальше *port — это int
```

**2. Monadic operations** (`and_then` / `or_else` / `transform`) — композиция операций с ошибками без if-цепочек:

```cpp
auto result = parse_port(argv[1])
    .and_then(open_socket)            // expected<Socket, string> ← expected<int, string>
    .and_then(start_listening)        // expected<Server, string> ← expected<Socket, string>
    .or_else([](const std::string& e) -> std::expected<Server, std::string> {
        std::println(stderr, "не удалось запустить сервер: {}", e);
        return std::unexpected(e);
    });
```

**3. Тестирование ошибочных веток — обязательный класс тестов.** Не «прогнал happy path, ОК». Полное покрытие включает:
- **Корнер-кейсы ввода:** пустой, граничный, превышающий лимит, неправильный формат, валидный по типу но запрещённый по семантике.
- **Fault injection** для IO/malloc/network: моки на стороне `std::fstream`, `LD_PRELOAD` для malloc, mock-server для сети.
- **Покрытие error path тестами** не ниже happy path — для финального проекта `gcov` / `Microsoft.CodeCoverage.Console` должен показать покрытие ≥80% не только успешных веток, но и `if (!result)` / `catch`-блоков.

**4. Исключения — ознакомительно.** Студент знает механизм, не использует как основную модель:
- **Поведение под `throw`:** stack unwinding, RAII-вызов деструкторов, поиск catch-блока по типу.
- **Exception safety guarantees:** basic (инвариант сохраняется), strong (commit-or-rollback), nothrow (`noexcept` гарантия).
- **`noexcept` спецификация** — обязательна на деструкторах, move-конструкторах, swap. Компилятор использует для оптимизации (например, `std::vector::push_back` move'ит только если move-конструктор noexcept).
- **Обёртки для C-ABI** — `extern "C"` функции должны быть noexcept, иначе UB при пересечении исключением C-границы:
  ```cpp
  extern "C" int my_callback(int x) noexcept {
      try {
          return process(x);  // может бросить
      } catch (...) {
          return -1;  // конверсия в C-стиль
      }
  }
  ```

**5. Stream-exceptions для IO-ошибок в декларативном pipeline.** Узкий специальный случай легитимного использования исключений в курсе — `ranges::copy(view, sink)` не имеет места для return-channel ошибки между итераторами:

```cpp
std::cin.exceptions(std::ios::badbit);   // ТОЛЬКО badbit, НЕ failbit
std::cout.exceptions(std::ios::badbit);

try {
    std::ranges::copy(
        std::views::istream<int>(std::cin) | std::views::filter(pred),
        std::ostream_iterator<int>{std::cout, "\n"});
} catch (const std::ios_base::failure& e) {
    auto ec = e.code();
    std::println(stderr, "не удалось обработать поток: {} (errno {})",
                 ec.message(), ec.value());
    return 74;
}
```

**Тонкость маски.** `failbit` срабатывает на EOF — для `views::istream<int>` EOF это **штатное завершение** (sentinel-итератор). Включить `failbit` в маску — pipeline сломается при нормальном конце ввода. `badbit` — только реальные IO-ошибки (broken pipe, нет места на диске, network mount исчез).

**Платформо-специфичность SIGPIPE.** `SIGPIPE` это POSIX-сигнал, на Windows его нет — `std::signal(SIGPIPE, SIG_IGN)` без обвязки не скомпилируется под MSVC. Корректно:

```cpp
#if defined(__unix__) || defined(__APPLE__)
#include <csignal>
namespace {
    struct IgnoreSigpipeInit {
        IgnoreSigpipeInit() { std::signal(SIGPIPE, SIG_IGN); }
    };
    const IgnoreSigpipeInit ignore_sigpipe_init;
}
#endif
```

Без `SIG_IGN` на Linux ядро убивает процесс мгновенно при попытке писать в закрытый pipe (`task | head`) — исключение не успевает броситься. На Windows `WriteFile` возвращает ошибку через `GetLastError()`, процесс не убивается.

##### Неделя 3 — локализация сообщений

Сообщения об ошибках выводятся на языке системной локали (минимум ru/en). Формат — GNU Coding Standards § 4.3 Errors:

```
progname: source-file:line: severity: message
```

Пример: `myutil: input.txt:42: error: invalid number 'abc'`. Полное предложение, заканчивается точкой, severity отдельным словом (не префиксом «Ошибка:»). Подробности стиля — § 2.1.C. Зрелый уровень локализации (`.po`/`.mo` файлы через gettext, перевод без перекомпиляции) — слой 25 в таблице § 2.1.A.

##### Неделя 4 — production-обработка

Цепочка: **разработка → пользователь → крах → восстановление**. Каждое звено бесполезно без остальных.

1. **Логирование с структурой** — каждое событие в JSON/structured form, с уникальным `event_id`, timestamps, контекстом операции, severity. Промышленный логгер (`spdlog` / `Boost.Log`) — см. строку «Протоколирование» в § 2.1.A.
2. **Split debug info** на Release-сборке — `objcopy --only-keep-debug binary binary.debug && strip --strip-debug binary` (Linux); MSVC `/Zi` + linker `/DEBUG` (Windows). Пользователю отдаётся stripped-бинарь; `.debug`/pdb архивируются по версии+SHA1 / build-id.
3. **Краш-менеджер** — Crashpad / Breakpad. При падении генерируется minidump + лог + (опц.) скриншот → отправляется на сервер разработчика. Без `.debug`/pdb дамп бесполезен (см. строку «Стратегия поставки» в § 2.1.A).
4. **Постмортем-анализ** — `gdb ./program core` с подгрузкой `.debug` (Linux); `cdb -z minidump.dmp -y "<symbol-path>"` + `!analyze -v` (Windows). Сопоставление с логом по timestamp восстанавливает контекст падения. Подробности — строки «Краш-менеджер» и «Постмортем-анализ дампа» в § 2.1.A.

##### Сводный список антипаттернов

Для быстрого ревью и code review-фикстур:

| Антипаттерн | Что не так | Правильно |
|---|---|---|
| `return 1;` везде | Недифференцированно, shell-обёртки не знают что чинить | sysexits 64/65/66/74; см. неделя 1 |
| `if (scanf(...) != 1) return 1;` | Не отличает EOF от не-числа от IO-ошибки | `cin.eof()` / `fail()` / `bad()`; `errno`/`feof`/`ferror` для C |
| `return ec.value();` (errno как exit) | Конфликт с зарезервированными 126/127/128+N; нестандартизировано между ОС | exit = категория sysexits; errno в stderr |
| `cin.exceptions(failbit \| badbit)` | `failbit` срабатывает на EOF — `views::istream` сломается при нормальном конце | Только `badbit` |
| `std::signal(SIGPIPE, SIG_IGN);` без `#ifdef` | На Windows не скомпилируется (`SIGPIPE` POSIX-only) | `#if defined(__unix__) \|\| defined(__APPLE__)` обвязка |
| `throw std::invalid_argument(...)` для парсинга argv | Исключения как control flow прикладной логики — анти-идиома в нашем стеке | `std::expected<int, E>` + `std::from_chars` |
| `e.what()` в сообщении без контекста | Штамп вида `"basic_ios::clear: iostream error"` — не информация | `ec.message()` + контекст операции (что делали в момент сбоя) |
| `std::println(stderr, "IO ошибка: {}", e.what());` | Префикс-штамп «ошибка» — нарушение § 2.1.C | `"не удалось обработать поток stdin: {} (errno {})", ec.message(), ec.value()` |
| `if (count == 0) return 1;` в функции + `if (... != 0) return 1;` в caller | Дублирование валидации; функция должна либо доверять caller (contract-based + `assert` в Debug), либо защищаться (defensive с осмысленным enum return type) | Выбрать одно: либо `@pre count > 0` в Doxygen + `assert(count > 0)` в Debug; либо `std::expected<T, EmptyInput>` с явным `if (!result) ...` в caller |
| `static_cast<int>(std::ssize(v))` для влива в `int count;` | Каст-симптом вместо лечения определения; обкаст не решает реальной проблемы | Изменить тип `count` на `std::size_t` / `std::ptrdiff_t` на месте определения (Core Guidelines ES.106/107) |
| `throw` через границу `extern "C"` | UB при пересечении исключением C-границы | `noexcept` обёртка с `try/catch (...)` внутри, конверсия в C-стиль return code |
| Тесты только на happy path | Покрытие ошибочных веток нулевое; регрессии в error handling не ловятся | Fault injection (IO/malloc/network), корнер-кейсы ввода, coverage ≥80% включая `if (!result)` / `catch` блоки |

##### Уровень оценки готовности к стажировке (расширение § 3 студенческих AGENTS.md)

Дополнительный критерий 8 для треков, использующих C++-стек: **дисциплина обработки ошибок** включает:
1. Дифференциацию exit-кодов по `sysexits.h` или собственному осмысленному enum, не `return 1;` везде.
2. Классификацию ошибок ввода (EOF / fail / bad для C++; `errno` / `feof` / `ferror` для C), различие в коде и в сообщении пользователю.
3. Двухканальный путь exit + stderr (категория для shell, детализация для человека); не миксует каналы (`return ec.value();` — антипаттерн).
4. Владение `std::expected<T, E>` (C++23) как первой моделью для прикладной логики; знание monadic operations.
5. Знание exception safety guarantees ознакомительно (basic / strong / nothrow); умение читать legacy-код с исключениями; обёртки для C-ABI границ.
6. Стиль диагностических сообщений § 2.1.C (конкретика, не штамп) в каждом `stderr`-сообщении / `throw` / `unexpected`.
7. Тестирование ошибочных веток как обязательный класс тестов; coverage error path не ниже happy path.

Это **редкая на собеседованиях компетенция**: программисты с алгоритмическим уклоном встречаются часто, программисты с дисциплиной обработки ошибок — редко. Отличительная черта курса.

### 2.2. Минимальный спектр для других треков
