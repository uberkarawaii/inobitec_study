## 4. Правила работы (важно)

### 4.1 Self-review перед коммитом

Перед каждым `git commit` / `git push`:

1. `git diff` пофайлово — пробежать глазами всё, что заливается.
2. Проверить: не залип ли мусор (артефакты сборки — `.obj`, `.exe`, `.pdb`, `.lib`, `.exp`, — временные файлы, логи прогонов, незавершённые комментарии, отладочные выводы, плейсхолдеры из ответов ИИ вроде «// ... существующий код ...»).
3. **Форматирование — механикой, не ручной дисциплиной.** Пока сборка ручная: `clang-format -i` на каждом изменённом файле + проверка `clang-format --dry-run -Werror` (падает на любом неотформатированном). Как только появляется `make` — обязательные цели `format` / `format-check`, и `test` не считается зелёным без формат-чистоты. `.clang-format` в корне репозитория — одна конфигурация на весь проект, **включая вспомогательные файлы** (генераторы, тесты): неважного кода в репо нет.
4. **Сборка с `/W4 /permissive-` зелёная — без warnings.** Дополнительно `/analyze` (статанализ MSVC) — каждое сообщение разбираем, не глушим.
5. **Тесты прогнаны в текущем рабочем режиме сборки** (§ 4.7). На 1-м этапе это Debug+ASan — единственная сборка. Если ASan ругается — фиксить, не заглушать.
6. Проект собирается с нуля одной командой (по мере лестницы: сценарий → `make` → `cmake --build`). **Вторая параллельная система сборки не заводится** — она неизбежно расходится с первой; переходный сценарий удаляется, когда появляется сборщик.
7. Тесты проходят и возвращают честный код: прогон обязан вернуть не-ноль, если упал хоть один кейс. Глотание ошибок в цикле прогона — дефект сборочной обвязки.
8. Все исправления по последнему ревью внесены. По каждому пункту ревью — короткий статус в переписке: **согласна/сделала · не согласна, потому что … · не поняла, вопрос**. Отклонение с аргументом — лучший ответ, чем молчаливое выполнение; молчаливый пропуск — худший.
9. **«Повторила дважды — выноси».** Если в diff один блок логики повторён два и более раз — перед коммитом вынести в функцию (или явно записать в коммит-сообщении, почему вынос сейчас неоправдан). Самопроверка: пробежать diff взглядом «что здесь похоже друг на друга?».
10. **Резервная копия кода — это git, не файл-копия.** Эксперимент — в ветке; состояние «до» — в теге или просто в истории. Закомментированные блоки «на всякий случай» и копии файлов рядом с проектом в коммит не идут: история и так всё хранит.

### 4.3 STL обязательна на C++-стороне

C++-реализация — **не «C с классами»** и **не «C с `std::print`»**, а идиоматический C++:

- **Контейнеры** (`std::vector`, `std::array`, `std::string`, `std::map`) вместо C-массивов, `char[]`, ручного `malloc`.
- **STL-алгоритмы** (`std::ranges::sort`, `minmax_element`, `fold_left`, `views::filter`) вместо ручных циклов с индексом.
- **RAII** — умные указатели, контейнеры — вместо ручных init/destroy.
- **Range-based for** и **structured bindings** вместо итераторов с индексом.
- **`std::string`, `std::format`, `std::print`/`std::println`** вместо `char[]` + `sprintf`.
- **Обработка ошибок — `std::expected<T, E>` (C++23) и коды как основа; исключения — ознакомительно** (архитектурное решение курса, обоснование — [02-praktika.md](02-praktika.md) § 2.1.F: тренд C++23/26, production-практика крупных проектов, специфика стека Inobitec).

Шпаргалка соответствий «C ↔ C++» — [02-praktika.md](02-praktika.md) § 2.1.B.

C-реализация и C++-реализация одной задачи — **разные**, не похожие. C-сторона — для понимания низкоуровневых механик (память, errno, ручные буферы); C++-сторона — то, как пишут production-код.

**Про runtime-проверки инвариантов:** `assert()` как рабочий инструмент не используем (выключается в Release, даёт ложное чувство безопасности). Вместо него: Debug+ASan-сборка (§ 4.7), контракты как документация (`@pre`/`@post` в Doxygen-комментариях), ошибки, работающие и в Release (`std::expected`, явные коды), юнит-тесты на корнер-кейсы.

#### 4.3.A Типовые ловушки «C++ как C»

Самый частый паттерн при фоне «C++ из универа»: взять C-логику, заменить вывод на `std::print` — и считать это C++. На ревью наставник ловит именно это:

| Паттерн «как в C» | Идиоматичный C++ | Почему |
|---|---|---|
| `(int)v.size()` — C-style cast | `static_cast<...>` или, лучше, подходящий тип (`std::size_t`) на месте определения | C-cast молча делает любое преобразование; а каст-симптом часто маскирует неверно выбранный тип |
| `INFINITY` (макрос `<math.h>`) | `std::numeric_limits<double>::infinity()` | макрос обходит систему типов |
| `char buf[256]; sprintf(buf, ...)` | `std::format(...)` / `std::println(...)` | buffer overflow при первой неосторожности |
| ручной `while`-цикл подсчёта разрядов числа | `std::formatted_size("{}", n)` | STL уже умеет длину форматированной строки |
| три прохода `min`/`max`/`sum` или ручной цикл-свёртка | `std::ranges::minmax_element` + `std::ranges::fold_left` | алгоритмы передают намерение и снимают классы ошибок инициализации свёртки целиком |
| `return std::string("лет");` из литерала | `std::string_view` | без аллокации на каждый вызов |
| `for (int i = 0; i < v.size(); ++i)` | `for (auto& x : v)` / ranges | индексный цикл — наследие C |
| `enum { RED, GREEN }` | `enum class Color { Red, Green }` | изоляция имён и типобезопасность |
| `#define MAX_N 100` | `constexpr int kMaxN = 100;` | проверяется компилятором, имеет scope |
| `void f(double* arr, size_t n)` | `void f(std::span<const double> arr)` | длина едет вместе с данными |
| обёртка `is_greater(x, k)` в своём модуле | `std::bind_back(std::greater{}, k)` | предикат и partial application уже в STL — модуль не нужен |

#### 4.3.B Типовые ловушки «C++ как C#»

При фоне C# перекос другой — и его наставник ловит так же. Главное отличие сред: в C# за временем жизни следит GC, в C++ — ты, через RAII; в C# всё — класс, в C++ класс нужен только там, где есть инвариант.

| Паттерн «как в C#» | Идиоматичный C++ | Почему |
|---|---|---|
| статический класс-хелпер на каждую группу функций | свободные функции в `namespace` | класс без состояния и инварианта — лишняя обёртка |
| проверка типа строкой (`GetType().Name == "..."`) | виртуальные функции / `std::variant` + `std::visit`; в крайнем случае `dynamic_cast` | строка не проверяется компилятором — опечатка молча ломает ветку |
| `try { ... } catch { }` «чтобы не упало» | узкий `catch` конкретного типа с обработкой, или не ловить вовсе; основная модель курса — `std::expected` | проглоченная ошибка всплывает дальше от причины и дороже |
| парсинг числа через подмену культуры (`Replace('.', ',')`) | `std::from_chars` | локале-независим по стандарту, без строковых трюков |
| `null`-проверки как основная защита | владение через типы: значение, ссылка, `std::optional`, умный указатель | система типов выражает «может отсутствовать» явно |
| глобальное статическое состояние между вызовами | состояние — в объекте, время жизни которого явно | у плагина/программы может быть два контекста сразу — статика молча затирается |

**Ритуал «после написания — проверить идиоматичность».** Когда версия работает, перед коммитом — запрос к DeepSeek:

> *Проверь этот код на C-наследие (C-cast, ручные циклы вместо алгоритмов, макросы вместо `constexpr`, ручные буферы вместо `std::format`) и на C#-привычки (статические хелперы вместо свободных функций, проверки типов строками, широкие catch). Покажи конкретные строки, предложи идиоматичную замену, объясни, почему она лучше.*

**Ритуал недели:** 1–2 видео C++ Weekly + одна идиома из них применена в код этапа. На контрольной точке наставник спросит: «какое видео смотрела, где применила?»

### 4.4 Параллельная реализация C и C++

Задачи 1–3 решаются **дважды**: на чистом C (C17) и на C++ (C++23). Цель — увидеть разницу подходов: `malloc/free` vs RAII; `errno` и коды vs `std::expected`; индексные циклы vs ranges; `printf` vs `std::print`; `char*` vs `std::string`.

Чистый C для тебя — **новый материал** (фон: C# и управляемый C++/CLI). Это нормально и учтено в темпе: C-сторона задач пойдёт медленнее C++-стороны, на ней закладывается понимание того, что C++ и тем более C# автоматизируют. Не проскакивай её через «DeepSeek, напиши C-версию» — именно тут видно усвоение.

### 4.5 Контрольные точки (по завершению этапа, очно)

Встреча с наставником ~2 часа, день договариваем гибко. Формат:

1. **Демонстрация результата этапа:** репозиторий, артефакт работает, тесты проходят.
2. **Работа на чистой виртуальной машине (Windows), которую даёт наставник:**
   - **Установка окружения с нуля** — VS Build Tools (или согласованный набор), FAR, git; `vcvars64`, проверка `cl`, ручная сборка hello world через `cl /c hello.cpp` → `link hello.obj`. Время этого этапа — само по себе показатель освоенности фундамента.
   - **Повтор ключевого куска задачи без ИИ + вариация от наставника.** Без OpenCode, без DeepSeek, без любых ИИ. **Интернет можно** — cppreference, Microsoft Learn, Stack Overflow. Наставник наблюдает за работой: команды, шорткаты, ориентация в FAR, чтение диагностики компилятора, выбор инструментов отладки.
3. **Краткое обсуждение** диалогов с DeepSeek за этап — где помог, где галлюцинировал, как ты это заметила.

**Это не экзамен.** Это active recall: если ты поняла свой код, написанный с ИИ, — ты его воспроизведёшь. Если прогнала через ИИ и приняла — повтор не пойдёт. Знание этого меняет твою работу весь этап: вникай, не делегируй слепо. Письменные ответы и пересказы каналом проверки не являются — только руки на чистой машине.

### 4.6 Стиль диагностических сообщений

Сообщение об ошибке — конкретика, не штамп: **что не получилось, какое значение получено, какое допустимо**. Без эмоций, без обвинений, без префиксов «Ошибка:», «Error:».

Плохо: `Ошибка, число не в диапазоне`. Плохо: `ошибка ввода: сбой потока`.
Хорошо: `N должно быть в [1; 20]; получено: 42`.
Хорошо: `не удалось прочитать stdin: <текст системной ошибки> (errno <код>)`.

Полное правило — [02-praktika.md](02-praktika.md) § 2.1.C. Применяется **к каждому диагностическому сообщению с первой задачи, включая новый код** — не только к местам, указанным в ревью. Самопроверка перед коммитом: `grep -nE "Ошибка|Error|Failed|Не удалось[^:]*$|сбой"` по diff'у — каждое совпадение пересмотреть.

Это не косметика: на ревью наставник смотрит сообщения об ошибках первым делом.

### 4.7 Режимы сборки (MSVC)

Сборка — последовательность подключения режимов по мере роста задачи, не «выбери один из четырёх». Полная матрица и соответствие GCC-стороне — [02-praktika.md](02-praktika.md) § 2.1.D.

**Ступень 1. Debug + ASan — стартовый и единственный режим 1-го этапа.**

Флаги: `cl /std:c++latest /W4 /permissive- /EHsc /Od /Zi /MDd /fsanitize=address`

- `/Od /Zi` — без оптимизаций, полная отладочная информация (pdb): отладчик идёт по строкам 1:1.
- `/fsanitize=address` — ASan: out-of-bounds, use-after-free; ловит баги в момент возникновения.
- `/MDd` — отладочный CRT + проверки итераторов STL (`_ITERATOR_DEBUG_LEVEL=2` по умолчанию).
- UBSan в MSVC отсутствует — классы UB частично закрывает `/analyze`; при необходимости подключим `clang-cl` (сверяться с § 2.1.D — состояние инструментов меняется).
- **Бенчмарки на Debug не делать** — медленнее в разы; замеры — только ступень 3.

**Ступень 2. Coverage — когда появились тесты.** Отдельная сборка и прогон через `Microsoft.CodeCoverage.Console`, отчёт с разбором непокрытых веток. Детали — на этапе 3.

**Ступень 3. Release — когда нужно мерить производительность.**

Флаги: `cl /O2 /Zi /DNDEBUG /MD` + линкер `/DEBUG /OPT:REF /OPT:ICF`.

**`/Zi` + `/DEBUG` обязательны и в Release — pdb всегда лежит рядом с exe.** Без pdb crash в production = голые адреса в стеке. pdb пользователю не отдаётся, но обязательно сохраняется.

**Ступень 4. Production — финальный артефакт.** Стратегия поставки CRT: `/MD` (динамический CRT, требует VC Redistributable на целевой машине) vs `/MT` (статический CRT, автономный exe большего размера) — это **часть проектного дизайна**. PGO для CPU-bound: `/GL` + `/LTCG:PGI` → нагрузочный прогон → `/LTCG:PGO`. Постмортем-цепочка: дамп (`procdump -ma`) → `cdb -z crash.dmp` + pdb → стек с именами.

**Ось конфигурации в сборке — закладывается заранее:** когда появится `make`/`CMake`, режим (Debug/Release/Coverage) — параметр сборки с раздельными каталогами артефактов, не правка флагов на месте. Иначе при вводе Release сборку придётся переписывать, а не расширять.

### 4.8 Спираль освоения: голая версия с первого дня, промышленная позже

Каждый инфраструктурный инструмент имеет **учебный голый аналог** из стандартной библиотеки или базового toolchain'а. Голая версия — **с первой задачи**; промышленная — когда задача доросла. Привычка пользоваться инструментом формируется в работе с ним, не «после установки настоящего».

| Тема | Голая (с 1-го этапа) | Промышленная (позже) | Когда переход |
|---|---|---|---|
| Юнит-тестирование | exe с `return 0/1` | GoogleTest | тестов 10+, нужны параметризованные |
| E2E CLI | `prog < in > out` + `fc`-сравнение с эталоном | матричный suite на GoogleTest + `boost::process` | сценариев 10+, интеграция в `ctest` |
| Логирование | `std::println(stderr, ...)` без обёрток | `spdlog` | нужны уровни/ротация/асинхронность |
| Версионирование | `kVersion` + `--version` + `CHANGELOG.md` | CMake `project(VERSION)` + git describe | появилась CMake-сборка |
| Сборка | `cl` руками → раздельные этапы | `.ps1` (переходно) → GNU make → CMake | второй модуль / надоело руками |
| Статанализ | `/W4 /permissive-` | `/analyze` → `clang-tidy` | сборка чиста по warnings |
| Форматирование | `.clang-format` с первого коммита | цели `format`/`format-check` в сборщике, CI | появился сборщик |

**Критерий лог-строки:** «что я захочу знать, читая этот лог после сбоя?» В лог идут факты, иначе не видимые: объёмы, длительности, переходы состояний. **Эхо пользовательского ввода — шум**; в коротких интерактивных программах логировать почти нечего, и это честный вывод, а не недоработка. Персональные данные в лог не пишутся (в медицинском ПО имя пациента в логе — инцидент): протоколируется факт события, не содержимое.

На ревью наставник смотрит: есть ли голая версия каждого инструмента уже на 1-м этапе. Промышленная версия позже **не компенсирует** отсутствие голой сейчас.

### 4.9 Политики сборки и тестов (действуют с первого дня)

1. **Одна точка входа в проект.** Вся цепочка «сборка → генерация входов → эталоны → прогоны → сравнение → формат» — воспроизводима одной командой. До сборщика — минимальный переходный сценарий; с появлением `make` — цели make с честными зависимостями, переходный сценарий удаляется. Проверка: на чистом клоне до зелёных тестов — одна команда, `make test`.
2. **Скриптовые полотна минимизированы.** Связка в одну строчку (в рецепте цели) — допустима; логика длиннее — пишется тестом на C/C++ (exe с `return 0/1`). Оркестрация на десятки строк в `.ps1`/`.bat` не заводится: она не видна ни компилятору, ни санитайзеру, и непереносима.
3. **Эталоны генерятся на лету, в репо не хранятся.** В репозитории — только исходники (генератор, программа, референс) и сборочные файлы. Эталон считает **независимая референс-реализация** (однострочник PowerShell / отдельный простой exe — не прогон тестируемой программы, иначе тест сравнивает её саму с собой). Замороженный снапшот выхода — не эталон корректности: он фиксирует и ошибку, если она была в первом прогоне. Сгенерированные данные живут в `build/` под `.gitignore`.
4. **Кроссплатформенность по умолчанию.** Платформо-специфичный код — в `#ifdef` (`_WIN32` / `__unix__` / `__APPLE__`); предпочитать стандартные альтернативы (`std::filesystem`, `std::chrono`, `std::format`, `std::thread`); типы фиксированной ширины (`int32_t`, `std::size_t`). **Сборочное описание — тоже кроссплатформенное**: Makefile пишется на GNU-make-синтаксисе с toolchain-различиями (MSVC / GCC) в условных блоках переменных, не двумя параллельными системами сборки (см. слой 15). Допустимо начать с одного заполненного toolchain-блока (MSVC) — с явной фиксацией в README, что GCC-блок не реализован.
5. **Магические числа именуются.** Числа с доменным смыслом — в именованные константы (`constexpr` в C++, `enum`/`#define` с префиксом проекта в C). Exit-коды — один раз в общем заголовке (`common/exit_codes.h(pp)`), не рассыпаны литералами. Обратный карго-культ (`kZero = 0`) не нужен. **Пути — никогда не абсолютные литералы**: рабочие файлы ищутся относительно явно определённой базы (аргумент, переменная окружения, каталог exe), и это решение фиксируется в README.
6. **Exit-коды дифференцированы с первой задачи.** Ориентир-семантика sysexits: 64 — usage (плохие аргументы), 65 — данные (не-число в потоке), 66 — нет входных данных, 74 — IO-сбой. На Windows `<sysexits.h>` нет — значения задаются своими константами в `common/`. **Двухканальность:** exit-код — категория для скрипта-обёртки (машинный канал), сообщение в stderr — детализация для человека (системный текст ошибки + контекст операции). Антипаттерн — `return errno;`.
7. **Тесты проверяют все три канала:** stdout (рабочий вывод), stderr (диагностика — не глушить `2>$null`, иначе штампы в сообщениях невидимы для тестов), exit-код (**именно ожидаемый**, не «не ноль» — иначе дифференциация отвалится молча). Ошибочные ветки (IO-сбой, пустой ввод) — тоже провоцируются тестами: код, который никогда не исполнялся, — код на доверии.

### 4.10 Диагностика до причины: обходной путь — не финал

Когда что-то «не работает, а вот так — работает», у расхождения есть причина, и она дешевле всего находится сейчас, пока контекст горячий. Правило курса:

**Процедура при любом «почему-то не работает»:**

1. **Воспроизвести минимально** — отрезать всё лишнее, пока сбой не останется в нескольких строках.
2. **Прочитать точный текст ошибки целиком** — код, имя файла, номер строки; не пересказ «что-то про линковку».
3. **Сформулировать гипотезу** — «думаю, дело в X, потому что Y» — и проверить именно её.
4. Только после этого — фикс. Фикс без подтверждённой гипотезы — это лотерея.

**Вопрос к ИИ — с точным текстом ошибки и вопросом «почему», не «сделай по-другому».** На «не работает, предложи другой способ» модель охотно выдаст обходной путь — и причина останется в коде миной. На «вот точная ошибка, вот минимальный пример — в чём причина?» — шанс понять.

**Обходной путь допустим только как явная временная заглушка:** рядом — комментарий `// TODO: обходной путь, причина не найдена; пробовала A и B, гипотеза C`, и запись в журнале (§ 7). Перед закрытием этапа каждый такой TODO либо заменён пониманием причины, либо вынесен вопросом наставнику. Типовые сигналы, что в коде поселился непонятый обходной путь: абсолютный путь «потому что иначе ошибка»; закомментированный блок «это не вышло»; подкручивание значений до совпадения; «работает — не трогаю».

**Таймбокс честных попыток.** Если после 3–4 итераций «гипотеза → проверка» причина не найдена — это не провал, а момент зафиксировать в журнале «что пробовала, что отбросила, где застряла» и спросить наставника. Вопрос с историей попыток — сильный ход, не слабость; молчаливый костыль — наоборот.

На контрольных точках наставник будет спрашивать «почему» по местам из твоего же кода — это проверяемая часть программы, наравне с тестами.

## 5. Лестница слоёв (Windows-flavor, ориентир)

Слои подключаются, когда задача доросла. **Не пропускаем слои**: если на текущем буксует — закрываем, не двигаемся дальше; перепрыгивание создаёт карго-культы (CMake поверх непонятого `cl /c` и `link`). Ссылки на слои — по имени. Развёрнутые описания и GCC-параллели — [02-praktika.md](02-praktika.md) § 2.1.A.

1. **Окружение и toolchain.** VS Build Tools, `vcvars64.bat` (или Native Tools Prompt — понять, что он делает с `PATH`/`INCLUDE`/`LIB`), FAR, git. Проверка: `cl` из консоли отдаёт версию.
2. **Репо-инфраструктура.** `git init` + `.gitignore` (`*.obj`, `*.exe`, `*.pdb`, `*.lib`, `*.dll`, `*.pch`, `build/`) + `.clang-format` + `.editorconfig` + минимальный `README.md` — **раньше первого hello world**: первый коммит формат-чистый. README — артефакт портфолио: описание, структура, как собрать, как прогнать тесты; поддерживается актуальным.
   - **Защита заголовочников — дифференцированно по языку:** `.h` (C) → `#ifndef`-guards с уникальным префиксом проекта (`IRINA_T1_GEOM_CORE_H`); `.hpp` (C++) → `#pragma once`. Единый стиль на каждый тип файла во всём репо.
   - **Порядок `#include`** (Google-стиль): свой header первым (тест self-containment) → пустая строка → C system → C++ stdlib → сторонние → проектные. Каждый `.cpp` явно включает всё, что использует, — не полагается на транзитивные.
3. **Hello world одной командой** — `cl hello.cpp` → `hello.exe`. Первый коммит.
4. **Базовый I/O.** `std::cin`/`cout`/`cerr`, `std::getline` (ловушка `\n` после `>>`), `cin.fail()`, чтение до EOF, `std::format`/`std::println`. Перенаправление PowerShell/cmd (`<`, `>`, `|`).
   - **Кодировка: стандарт слоя — CP1251.** Исходники с русскими литералами сохранять в CP1251 (FAR Editor умеет), консоль — `chcp 1251`, в коде `setlocale(LC_ALL, "Russian_Russia.1251")` первой строкой `main()` — тогда `<ctype.h>` корректен на русских буквах, `name[i]` — символ. **`chcp 65001` (UTF-8) для интерактивного ввода русских букв нестабилен** (известный дефект моста CRT↔консоль: pipe-тесты проходят, живой ввод теряется) — не использовать до слоя локализации. `setlocale` ставится осознанно: убери его и посмотри, что изменилось, — если ничего, значит в этом месте он не нужен. **Кодировка всех исходников репозитория — одна, зафиксирована в `.editorconfig`**; смесь CP1251/UTF-8 по файлам — дефект репо-инфраструктуры, ловится на ревью.
5. **Базовые ключи и warnings.** `/W4 /permissive- /std:c++latest /EHsc /Od /Zi`. Каждое предупреждение разбираем.
6. **Тестирование без фреймворка.** (6a) юнит-тест функции = отдельный exe с `return 0/1`; (6b) acceptance: `prog < test_data.txt > out.txt` + `fc out.txt etalon.txt` — с учётом политик § 4.9 (эталон на лету, проверка stderr и exit-кода).
7. **Отладочная печать.** `std::println(stderr, "x={}", x)` — временный инструмент, удаляется перед коммитом (в отличие от протоколирования).
8. **Отладчик `cdb`.** Breakpoints, step, стек, перенаправление stdin при запуске под отладчиком. Уровни владения: запуск с `< file` → отладка с конвейерным входом → attach по PID. Если при баге первым делом тянешься к `printf` — отладчик ещё не освоен.
9. **Раздельные этапы сборки.** `cl /c file.cpp` → `.obj`; `link main.obj foo.obj /OUT:prog.exe`; `lib /OUT:mylib.lib *.obj` (статическая); **`cl /LD utils.cpp` → `.dll` + import-`.lib`, линковка потребителя через import-lib** — обязательный пункт фундамента, не опция. Подключается, когда у задачи появился второй модуль. *(Здесь же раз и навсегда проясняется, что такое `.obj`, `.pdb`, `.lib`, `.exp`, `.ilk` — и почему всем им место в `.gitignore`, а не в репозитории.)*
10. **UX и справка CLI.** `argc`/`argv`, `--help`, `--version`, разделение stdout/stderr, дифференцированные exit-коды (§ 4.9 п. 6), pipe-friendly поведение.
11. **Голое протоколирование** (§ 4.8, критерий лог-строки там же).
12. **Голое версионирование.** `kVersion` + `--version` + `CHANGELOG.md` (Keep a Changelog).
13. **Пакетный сценарий `.ps1`** — переходный слой: когда `cl` руками надоел. Живёт до сборщика, удаляется при появлении `make`.
14. **PCH.** MSVC-механика: `/Yc"pch.h"` (создать) / `/Yu"pch.h"` (использовать) — stdafx-стиль; либо force-include `/FI`. **Две когерентные схемы**: или pch включается первой строкой и дубли заголовков убраны, или pch только в флагах сборки и исходники держат реальные зависимости. Гибрид «включаю pch И дублирую те же заголовки» бессмыслен. В pch — только стабильные тяжёлые заголовки (STL, SDK), не свои часто-правимые: иначе каскадная пересборка. Польза измеряется секундомером (`Measure-Command` на полной пересборке с pch и без), не предполагается.
15. **Сборщик GNU make.** На Windows ставится `winget install ezwinports.make` (4.4+), запускается из того же Native Tools-окружения, что и `cl`. Родной для MSVC `nmake` в курсе не используем: его синтаксис (без pattern rules, свои инференс-правила) никуда не переносится, а GNU-make-синтаксис — индустриальный стандарт. Главный смысл слоя — **граф зависимостей как модель**: цель → пререквизиты → рецепт, пересборка только изменившегося; pattern rules (`build/%.obj: %.cpp`), переменные, `.PHONY`, order-only пререквизиты. **Данные тестов — тоже узлы графа** (вход — цель от генератора, выход — цель от входа и бинаря), не побочный продукт скрипта.
    - **Кроссплатформенность Makefile решается на этом слое, а не откладывается до CMake.** Один Makefile, граф и цели общие; toolchain-различия — в условных блоках переменных (`ifeq`): компилятор (`cl` / `g++`), флаги (`/W4 /EHsc /Fo` vs `-Wall -o`), суффиксы (`.obj`/`.exe` vs `.o`/пусто), команды сравнения (`fc` / `diff`). Рецепты — переносимые однострочники через эти переменные, без shell-измов (логика длиннее однострочника — это тест-exe, § 4.9 п. 2). Проверка слоя: тот же Makefile с заполненным GCC-блоком собрал бы проект на Linux.
    - **Header-зависимости:** на GCC-стороне это автоматика `-MMD -MP`; у `cl` аналога нет — есть `/showIncludes` (парсинг его вывода — продвинутое упражнение). На учебном масштабе деп-листы на MSVC ведутся руками — и протухающая ручная зависимость, которую забыли обновить, — честная мотивация следующего слоя.
16. **Метасборщик CMake.** `CMakeLists.txt` → `cmake -B build` → `cmake --build build`. Мотивация после переносимого Makefile — не «переносимость» (она уже есть), а: автоматические header-зависимости (то, что на слое 15 делалось руками), абстракция toolchain-флагов (ifeq-блоки исчезают), `find_package` + vcpkg для зависимостей, `target_precompile_headers`, генерация под разные сборщики и IDE.
17. **E2E с генератором.** `gen_N.cpp` (`std::mt19937` с фиксированным seed) для массовых входов; эталон — независимый референс (§ 4.9 п. 3).
18. **Пакетный менеджер `vcpkg`** (manifest-mode). Первая библиотека — собрать один раз вручную, чтобы увидеть, что vcpkg автоматизирует.
19. **GoogleTest** → 20. **Матричный CLI-acceptance (GoogleTest + `boost::process`)** — инфраструктура тестирования строится полностью **до** оптимизаций: меняя алгоритм, прогоняем suite и видим, не сломалось ли поведение; без полной testbed оптимизация — слепая работа → 21. **Покрытие** (`Microsoft.CodeCoverage.Console`) → 22–23. **Статанализ** (`/analyze`, `clang-tidy`) → 24. **Doxygen** → 25. **Локализация** (ru/en по локали; здесь — момент пересмотра кодировки).
26. **Оценка алгоритмической сложности и адаптация под ресурсы — рефакторинг производительности.** Лестница «корректно → масштабируемо → быстро алгоритмически → быстро параллельно». Big-O-анализ; выбор алгоритма под сценарий-ограничение («мало памяти / много данных / слабый CPU» — переключатель алгоритма, не просто «оптимизируй»); замеры `std::chrono` **только на Release** (§ 4.7 ступень 3 — на этом слое Release появляется впервые). Обязательный артефакт — **таблица замеров `N → время` по всем вариантам лестницы**: где какой вариант начинает выигрывать и почему; расхождение с теорией — повод искать причину, не игнорировать. Дисциплина **тестово-управляемой оптимизации**: меняем алгоритм → тестовый suite зелёный (поведение не изменилось) → замер → решение оставить/откатить. Цель `bench` в Makefile: большой вход от генератора (`--size`/`--seed`) + замер времени = реперная точка до и после каждого шага. Решение об оптимизации принимается **по замерам на конкретном профиле нагрузки**, не «интуитивно по виду кода»; эквивалентный по логике код может иметь разную скорость из-за микроархитектуры CPU.
27. **Параллелизм — глобальный навык, последняя ступень лестницы оптимизаций** (после корректной и алгоритмически быстрой версии — параллелить медленный алгоритм бессмысленно). Прогрессия: **Parallel STL** (`std::execution::par` в `std::sort`/`for_each`/`reduce` — одна строка, MSVC поддерживает из коробки) → **`std::thread` / `std::jthread` / `std::async`** (ручное разбиение работы, агрегация результатов) → **OpenMP** (`#pragma omp parallel for`, в MSVC — ключ `/openmp`) → SIMD (опционально). Обязательное понимание: data races, deadlocks, false sharing; когда параллелизм **не** ускоряет (IO-bound, мелкие задачи — оверхед запуска потоков съедает выигрыш). Каждая параллельная версия — строка в таблице замеров слоя 26 рядом с последовательной; для многопоточного кода — TSan-класс проверок (на MSVC ASan гонки не ловит — анализ через `/analyze` и дисциплину, на CMake-слое возможен прогон под clang).
28. **PGO** (§ 4.7 ступень 4): `/GL /LTCG:PGI` → нагрузочный прогон на репрезентативном входе → `/LTCG:PGO`; 5–15 % на CPU-bound «бесплатно», на IO-bound прироста нет — и это тоже проверяется замером, не предполагается.
29. **Пакет утилит с subcommands.**

Блок production-эксплуатации (промышленное логирование, версионирование через CMake, стратегия поставки CRT + pdb, краш-менеджер, постмортем `cdb -z`) — карта роста после оценочного периода, в этапы 1–4 не входит.

### 6.1 Базовый набор: геометрия точек — ввод/вывод и аргументы командной строки

Сквозной домен задачника — **точки в 3D** (`x y z`, вещественные). Каждая задача решается дважды — C17 и C++23 (MSVC, `/std:c17` / `/std:c++latest`), режим сборки — Debug+ASan (§ 4.7 ступень 1). Сборка пока вручную: `cl /c` → `link`.

**Сколько успеть на 1-м этапе:** базовая планка — 3 задачи, хорошая — 4, отличная — 4 + полная тестовая цепочка к каждой. Темп определит план дальше; буксование на слое — сигнал закрыть слой, не провал.

**Задача 1. Матрица попарных расстояний.** Дано целое `N` (3 ≤ N ≤ 20). Построить вершины правильного `N`-угольника единичного радиуса с центром в начале координат (в плоскости `z = 0`) и вывести матрицу `N×N` попарных расстояний между вершинами, выровненную по столбцам, с 3 знаками после запятой. Диагональ — нули, матрица симметрична. `N` вне диапазона или не число → диагностика по § 4.6 в stderr, ненулевой код возврата.

**Задача 2. Паспорт фигуры.** Программа запрашивает название фигуры (строка, русские буквы — может быть из нескольких слов) и число вершин `V`, выводит «Фигура «звезда»: 7 вершин.» с правильным склонением **«вершина / вершины / вершин»** по последней цифре. *(Первое столкновение с кодировкой: исходник в CP1251, консоль `chcp 1251`, `setlocale` — см. слой 4. Пустое имя и EOF — два разных случая с разной диагностикой.)*

**Задача 3. Bounding box и центроид облака точек.** Читает из stdin точки `x y z` (по одной в строке) до EOF. Выводит: количество точек, min/max по каждой из трёх осей (ограничивающий параллелепипед), центроид, среднее расстояние от точек до центроида (3 знака). Корректно работает через перенаправление: `prog < cloud.txt > report.txt`. Классы ошибок различаются по § 4.9 п. 6: не-число посреди потока (65) / пустой ввод (66) / сбой IO (74) — с конкретной диагностикой (номер битой строки — в сообщение).

**Задача 4. Фильтр точек по радиусу.** Первым аргументом — вещественное `R > 0`. Из stdin читает точки до EOF, в stdout выводит только те, чьё расстояние от начала координат меньше `R`, по одной в строке, в том же формате `x y z`. Аргумент не задан / не число / не положителен → код 64 с диагностикой по § 4.6 в stderr.

**К каждой задаче — тестовая цепочка** (без неё задача не завершена, даже если работает на ручных входах), по политикам § 4.9:

- **Acceptance (6b, все задачи):** вход — от генератора или короткий литерал; эталон — независимая референс-реализация на лету (для задачи 1 — формула длины хорды `2·sin(π·k/N)`; для задач 3–4 — однострочник PowerShell или отдельный простой exe-референс); сравнение `fc`; проверка stdout, stderr и **конкретного** exit-кода. Корнер-кейсы (пустой ввод, граница диапазона, не-число, точка ровно на радиусе `R`) — отдельными кейсами.
- **Массовый вход (задачи 3–4):** генератор облака точек `gen_cloud --size --seed` (`std::mt19937`, фиксированный seed), размеры 10 / 100 / 10⁶; данные в `build/` под `.gitignore`.
- **Юнит-уровень (6a):** чистая логика — склонение (`vertex_word`), геометрическое ядро (расстояние, центроид, вершина N-угольника), парсинг `R` — exe с `return 0/1`.

**Сравнение вещественных в тестах:** расстояния и центроиды — float-арифметика; эталон и результат сравниваются с фиксированным форматом вывода (3 знака — тогда `fc` честен) либо с допуском в exe-тесте. Побайтовое сравнение неокруглённых double — ловушка, не делать.

**Что осваивается по ходу:** потоки и форматирование, ловушка `\n` после `>>`, цикл до EOF, перенаправление (увидеть: программа не знает про файлы — это shell их подсунул), `argc`/`argv`, парсинг строки в число (`std::from_chars` → `std::expected` как целевая идиома), `std::isfinite` и дисциплина работы с вещественными, коды возврата, stderr.

**Задел на этапы вперёд:** общая геометрическая логика задач 3–4 (точка, парсинг строки `x y z`, расстояние) — кандидат на общий модуль при появлении реальной общей логики (слой 9; обёртка над одной строкой STL модулем не считается — § 4.2, трёхступенчатая проверка). На этапе 2 эти же задачи — материал лестницы оптимизаций (попарные расстояния O(N²) → пространственная сетка → параллельно); на этапе 4 — пакет `geom` с subcommands.

**Что НЕ подключается сейчас** (рано): `fopen`/`ifstream` (хватает перенаправления), CLI11, GoogleTest, CMake, vcpkg.

## 8. Диалоги с DeepSeek

**Что:** полный машинный экспорт переписки с DeepSeek через OpenCode — JSON со всем содержимым сессии: реплики, рассуждения, **полный вызов каждого инструмента**, временные метки. **Без обработки, без выжимок, без редактуры, без ручной транскрипции** — обработка разрушает сигнал, который наставник в логе ищет.

**Зачем:** наставник смотрит манеру общения с ИИ — какие вопросы задаёшь, замечаешь ли галлюцинации, применяешь ли анти-подхалимажные приёмы. По манере видно: ты извлекаешь навык, или ИИ делает работу за тебя. Это основной канал, по которому наставник даёт рекомендации.

**Как выгружать — штатная команда OpenCode, руками (не просить DeepSeek сделать за тебя):**

```powershell
opencode session list                                  # найти ID сессии
opencode export <session-id> > dialog_logs/2026-06-15_task1.json
```

Каталог `dialog_logs/` — в корне репо, коммитится вместе с кодом. **В именах файлов не использовать `:`** (NTFS не примет у получателя) — дата и тема через `_` и `-`.

**Когда выгружать — СРАЗУ после сессии**, не в конце недели: OpenCode автоматически чистит старые тела сообщений; не выгрузил сразу — экспорт может вернуть пусто. **Дисциплина сессий: новая задача/тема = новая сессия**, не одна накопительная на месяц — каждый лог про одну тему.

**Что НЕ делать:** пересказы, конспекты, рефлексию пост-фактум, редактуру JSON. Опечатки и неудачные попытки — то самое сырьё. Анализ — работа наставника; твоя рефлексия встроена в сам диалог (приёмы § 4.2) в момент его ведения.
